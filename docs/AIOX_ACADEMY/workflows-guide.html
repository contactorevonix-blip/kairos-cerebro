📝 Story Development Cycle (SDC) — 5 Phases

The primary workflow for all feature implementation. Covers create → validate → implement → QA → push.

Phase 1
CREATE (@sm — River)
Create a new story from epic context and requirements.
Input: PRD + Epic | Output: story.md (Draft) | Time: 30 min - 2h
📌 Example
@sm reads: "Implement email notifications"
Creates: docs/stories/epics/1/1.2.story.md
Status: Draft
Ready for: @po validation
Phase 2
VALIDATE (@po — Pax)
10-point checklist: AC clarity, scope, dependencies, etc.
Input: story.md | Output: GO/NO-GO + Checklist | Time: 15-30 min | Gate: MUST PASS
✓ Validation Checklist
1. AC clarity (clear? testable?)
2. Scope fit (fits epic?)
3. AC count (3-5 ACs?)
4. Dependencies mapped
5-10. [Others...]

Result: GO ✓ → proceed to phase 3
Phase 3
IMPLEMENT (@dev — Dex)
Code implementation following acceptance criteria.
Input: Validated story | Output: Code + File List | Status: InProgress
💻 Developer Workflow
1. git checkout -b feature/email-notifications
2. Implement ACs
3. npm run lint (pass)
4. npm run typecheck (pass)
5. npm test (pass)
6. Self-critique
7. git commit -m "feat: email notifications [1.2]"
8. Request @qa review
Phase 4
QA GATE (@qa — Quinn)
7 mandatory quality checks before merge.
Input: Code | Output: PASS/CONCERNS/FAIL | Gate: MUST PASS | Max Iterations: 5
🧪 Quality Checks
1. npm run lint ✓
2. npm run typecheck ✓
3. npm test ✓
4. npm run build ✓
5. CodeRabbit (no CRITICAL) ✓
6. No secrets in code ✓
7. Story status = Done ✓

Verdict: PASS ✓
Phase 5
PUSH (@devops — Gage)
Deploy to main. EXCLUSIVE operation.
Input: Approved code | Output: Commit on main | Authority: @devops ONLY
🚀 Push Process
@devops executes:
git push origin feature/... → main

Result: Code shipped to production ✓
🎯 When to Use SDC
Bug fix or config change → Quick Flow variant
New feature (1-3 stories) → Standard SDC
Estimated < 2 days → SDC without Spec Pipeline
Complex feature → Spec Pipeline FIRST, then SDC

🔬 Spec Pipeline — 6 Phases

Pre-implementation phase for complex features. Transforms requirements into executable specs.

Phase 1-2
Gather & Assess
@pm gathers requirements. @architect assesses complexity.
Output: requirements.json + complexity.json
Phase 3
Research (@analyst)
Deep research if STANDARD/COMPLEX class.
Skipped if SIMPLE class
Phase 4
Write Spec (@pm)
Create spec.md with all requirements mapped to FR-*/NFR-*
Constitution Article IV: ZERO invention
Phase 5
Critique (@qa)
QA reviews spec for completeness and correctness.
Verdict: APPROVED | NEEDS_REVISION | BLOCKED
Phase 6
Plan (@architect)
Implementation plan if APPROVED.
Then → proceed to SDC for each story
🎯 When to Use Spec Pipeline
Complex feature (>15 stories) → REQUIRED
New product/epic → REQUIRED
Uncertain scope → Use Spec Pipeline
Simple bug fix → Skip Spec Pipeline

🔁 QA Loop — Iterative Review-Fix

Automated review-fix cycle when QA finds issues.

Cycle
Review → Fix → Re-Review (max 5 iterations)
@qa reviews code → verdict → @dev fixes → repeat
Config: max 5 iterations | Escalate after max reached
📋 Example Loop
Iteration 1:
@qa: "CONCERNS - missing test coverage"
@dev: Adds tests → commits

Iteration 2:
@qa: "PASS ✓"
Result: Code shipped
🎯 Verdicts
PASS ✓ → Code ships (complete)
CONCERNS → @dev fixes (continue loop)
FAIL → Critical issue (return to Phase 3)
BLOCKED → Escalate immediately

🏗️ Brownfield Discovery — 10 Phases

Technical debt assessment for legacy codebases.

Phases 1-3
Data Collection
@architect, @data-engineer, @ux audit existing system
Output: system-architecture.md, SCHEMA.md, frontend-spec.md
Phases 4-7
Draft & Validation
@architect drafts tech debt. @data-engineer + @ux review.
Gate: APPROVED | NEEDS_WORK
Phases 8-10
Finalization
Final assessment + executive report + ready-to-develop epics
Output: technical-debt-assessment.md + TECHNICAL-DEBT-REPORT.md
