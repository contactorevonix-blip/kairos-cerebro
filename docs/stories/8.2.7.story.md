# Story 8.2.7: IDS Health Dashboard

**Status:** Ready
**Story ID:** 8.2.7
**Created:** 2026-06-11
**Epic:** EPIC-8 — Framework Evolution (Phase 2: IDS Enhancement)
**PRD Source:** `docs/prd/epic-8/phase-2-ids.md#827-ids-health-dashboard-2sp`
**Story Points:** 2

---

## Story

**As a** framework maintainer or PM,
**I want** a web dashboard at `/ids` showing registry health, REUSE/ADAPT/CREATE distribution, gate compliance trends, circular dependencies, and high-risk entities,
**so that** I can see at a glance whether the framework's pattern-reuse discipline (Article IV-A) is improving or degrading over time, without manually querying the registry, gate logs, and changelog separately.

---

## Acceptance Criteria

1. Dashboard URL: `http://localhost:3000/ids`
2. Metrics displayed:
   - Registry health score (0-100)
   - REUSE % (entities reused / total entities)
   - Adaptation distribution (pie chart: REUSE vs ADAPT vs CREATE)
   - Gate compliance: G1-G6 pass rate by week
   - Circular dependencies detected
   - High-risk entities (score <0.3)
3. Drill-down: Click entity → show adaptability score + history
4. Export: Dashboard data exportable as JSON/CSV
5. Update frequency: Real-time (Webhook from gate events)

## Risks

- **Risk:** This is the highest-integration story in Phase 2 — it depends on 6 other stories, several of which (8.2.1-8.2.6) have open architecture questions of their own (CLI vs API hosting model for 8.2.4, registry write centralization for 8.2.6, etc.). If those decisions land inconsistently, this dashboard has to reconcile multiple data shapes.
- **Mitigation:** This story should NOT begin implementation until 8.2.1-8.2.6 are at least in `Ready` status with their architecture decisions resolved (especially 8.2.4's API/CLI hosting decision, since AC1 requires `http://localhost:3000/ids` — implying a web server already exists or must be created here).

- **Risk:** AC1 specifies `http://localhost:3000/ids` — port 3000 is a common default for Next.js dev servers. No existing web dashboard server was found referenced in this story's research materials.
- **Mitigation:** @architect must confirm whether port 3000 implies reuse of an existing project web app (e.g., the `nextjs-react` active preset per `.aiox-core/data/technical-preferences.md`), or a new standalone dashboard server. Per Article IV-A, REUSE the existing Next.js app structure if `kairoscheck.net` or another project app already runs on a compatible stack — do not assume a brand-new server without checking.

- **Risk:** AC5 "Real-time (Webhook from gate events)" implies gate hooks (`.claude/hooks/enforce-*.cjs`) need to emit webhook calls — these currently write to local JSONL files (`.aiox/gate-logs/`), not network calls.
- **Mitigation:** "Real-time via webhook" may need to be interpreted pragmatically as "near-real-time via file-watch + polling" given the current gate architecture (file-based JSONL logs, graceful-degradation philosophy). If true webhooks are required, this is a larger architectural addition to the gate hooks themselves — flag for @architect scope discussion before committing to the literal AC5 wording.

### Architecture Notes

- No specific guidance found in `docs/architecture/` for a web dashboard server. The `aiox graph --format=html` capability (`.aiox-core/core/graph-dashboard/`) is the only existing "dashboard-like" precedent in the AIOX core and should be evaluated first per Article IV-A REUSE>ADAPT>CREATE before building a new Next.js route.
- Per Article I (CLI First, NON-NEGOTIABLE), if a CLI alternative makes sense (e.g., `aiox ids dashboard` opening a local HTML report similar to `aiox graph --format=html`), that may better satisfy the framework's CLI-first principle than a persistent `localhost:3000` server — raise with @architect during Task 1.

### Testing

- Data aggregation test: for each of the 6 metrics in AC2, verify the computed value against a known/synthetic dataset before wiring to live data.
- Drill-down test: select a known entity (e.g., `add-mcp`, `usedBy: [devops]`) and verify the displayed score + history matches 8.2.4/8.2.6 outputs.
- Export test: JSON and CSV exports must contain the same underlying data as the rendered dashboard (no fields silently dropped).
- Accessibility test: run an automated a11y check (e.g., axe) against the `/ids` page if a testing framework with a11y support exists in the project.
- Update mechanism test: verify the chosen AC5 implementation (webhook/polling/file-watch) actually reflects new gate-log entries within a reasonable, documented latency.
