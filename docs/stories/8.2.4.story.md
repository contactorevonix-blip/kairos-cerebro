# Story 8.2.4: Adaptability Scoring

**Status:** Ready
**Story ID:** 8.2.4
**Created:** 2026-06-11
**Epic:** EPIC-8 — Framework Evolution (Phase 2: IDS Enhancement)
**PRD Source:** `docs/prd/epic-8/phase-2-ids.md#824-adaptability-scoring-2sp`
**Story Points:** 2

---

## Story

**As a** PM or developer deciding between REUSE, ADAPT, or CREATE for a new piece of work,
**I want** every entity in the registry to have a documented, formula-based adaptability score (0-1),
**so that** the REUSE>ADAPT>CREATE decision (Article IV-A) is backed by a transparent, reproducible metric instead of an undocumented `adaptability.score` field that nobody can explain or recompute.

---

## Acceptance Criteria

1. Scoring model: `docs/architecture/ids-adaptability-score.md`
   - Reusability (0-1): How often reused? How generic?
   - Clarity (0-1): Documentation quality? Type hints?
   - Flexibility (0-1): How easy to adapt? Configurable parameters?
   - Overall Score = 0.4*reuse + 0.3*clarity + 0.3*flexibility
2. API: `GET /api/ids/entity/{id}/score` (REST endpoint)
3. Dashboard widget: Show top-5 highest-score entities per category
4. Alerts: Entities with score <0.3 flagged for refactor (Art. IV)

## Risks

- **Risk:** `entity-registry.yaml` already has an `adaptability.score` field populated for all 823 entities (e.g., `add-mcp: 0.8`, `advanced-elicitation: 0.8`), but the research doc states "No clear algorithm for assigning/updating scores" — existing scores may not match the new formula in AC1.
- **Mitigation:** This story defines and documents the formula (AC1) and a recompute path. Whether to mass-recompute all 823 existing scores is a scope decision for @dev/@architect — at minimum, the new formula must be documented and a recompute mechanism must exist (even if run as a one-time migration script). Do not silently leave stale scores undocumented as "the new formula" without a recompute plan.

- **Risk:** AC2 introduces a REST API (`GET /api/ids/entity/{id}/score`) — no existing API server/framework was found referenced in this story's research for the AIOX core tooling (which is primarily CLI-based per Article I "CLI First").
- **Mitigation:** Per Article I (CLI First, NON-NEGOTIABLE), confirm with @architect whether AC2 should be a CLI command (`aiox ids score {id}`) exposing the same data, with the REST endpoint as an optional wrapper, or whether a lightweight API server already exists (e.g., for the IDS Health Dashboard in 8.2.7) that this can attach to. Do not invent a new API framework without architectural sign-off.

## Tasks / Subtasks

- [ ] Task 1 (AC: 1) — Author `docs/architecture/ids-adaptability-score.md`
  - [ ] Document the 3 sub-scores (Reusability, Clarity, Flexibility) with concrete, computable definitions:
    - Reusability: derive from `usedBy[].length` (more consumers = higher reuse) and entity `type` (per research doc: agents 0.0-0.3, tasks 0.7-1.0)
    - Clarity: derive from presence/completeness of `purpose`, `keywords`, documentation references
    - Flexibility: derive from `adaptability.constraints[].length` (fewer constraints = more flexible) and `adaptability.extensionPoints[].length` (more extension points = more flexible)
  - [ ] Document the Overall Score formula exactly: `Overall Score = 0.4*reuse + 0.3*clarity + 0.3*flexibility`
  - [ ] Cross-reference the research doc's proposed factors (`docs/research/ids-enhancement-gaps.md#gap-4-adaptability-scoring-logic-unclear`) — Type, Consumers, Stability, Dependencies — and reconcile with the 3-factor PRD formula (the research doc's factor list informs how to compute Reusability/Clarity/Flexibility, but the PRD's 3-term weighted formula is the binding AC)
- [ ] Task 2 (AC: 2) — Implement `GET /api/ids/entity/{id}/score`
  - [ ] **Architecture decision required first** (see Risks): confirm hosting model with @architect before implementation — CLI command (`aiox ids score {id}`), attach to an existing API server, or new lightweight server tied to 8.2.7's dashboard
  - [ ] Implement score computation per the AC1 formula, reading from `entity-registry.yaml`
  - [ ] Return the 3 sub-scores plus the overall score in the response (for transparency/debuggability)
- [ ] Task 3 (AC: 3) — Dashboard widget data
  - [ ] Compute "top-5 highest-score entities per category" (categories = the 7 entity types: tasks, templates, scripts, modules, agents, checklists, data per `entity-registry.yaml` metadata)
  - [ ] Expose this as a data structure consumable by Story 8.2.7's dashboard (coordinate field naming with 8.2.7 when that story is implemented)
- [ ] Task 4 (AC: 4) — Low-score alerting
  - [ ] Identify all entities with overall score <0.3
  - [ ] Document these as "flagged for refactor" per Article IV — output a report (e.g., `aiox ids score --alerts` or similar) listing flagged entity IDs, paths, and scores
  - [ ] Do NOT auto-refactor flagged entities — this story only flags them
