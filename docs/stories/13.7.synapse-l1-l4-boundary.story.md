## Story Statement

**As a** framework architect,  
**I want** a clear definition of Synapse responsibilities split between L1 (engine code) and L4 (runtime state),  
**so that** developers don't mix concerns and future modifications respect the boundary.

---

## Problem

Synapse engine is split:
- **L1:** `.aiox-core/core/synapse/` — motor code (engine, context-engine, manifests)
- **L4:** `.synapse/` — runtime state (caches, sessions, metrics, hooks)

**Risk:** Unclear who owns what. Developers might put logic in L4 or state in L1. Inverted dependencies (L4 importing L1 logic).

**Impact:** Modifications are risky; dependencies get tangled.

## Technical Context

### L1: Synapse Engine Code

**Location:** `.aiox-core/core/synapse/`

**Expected Contents:**
- `engine.js` — main Synapse execution engine
- `context-engine.js` — context management
- `manifest.js` — manifest parsing
- `manifests/` — built-in manifest definitions
- Interfaces/contracts (abstract classes, type definitions)

**Responsibility:**
- Define what Synapse does (rules evaluation, context evaluation)
- Export APIs for L4 to call
- NOT store state (state is L4 responsibility)
- NOT depend on specific L4 implementations

### L4: Synapse Runtime State

**Location:** `.synapse/`

**Expected Contents:**
- `caches/` — performance caches (generated from L1 rules)
- `sessions/` — session state
- `metrics/` — performance/execution metrics
- `hooks/` — registered hook callbacks
- `config.yaml` — runtime configuration

**Responsibility:**
- Store execution state
- Call L1 APIs to evaluate rules
- Track metrics/telemetry
- NOT define Synapse logic (that's L1)

### Inverted Dependency Pattern

**Question:** Do L4 modules import from L1?

**Scenario 1 (Normal):**
```javascript
// L1 exports API
module.exports = { evaluateRules, getContext }

// L4 imports and uses
const synapse = require('./../.aiox-core/core/synapse/engine')
synapse.evaluateRules(rules, context)
```

**Scenario 2 (Inverted — might be issue):**
```javascript
// L4 has state; L1 reads it directly
const stateFile = './../.synapse/sessions/active.json'
// L1 should NOT import state directly
// Instead, L1 should receive state as parameter
```

**Best Practice:**
- L1 defines interfaces
- L4 implements interfaces and stores state
- L1 never reads L4 files directly (only receives data as function parameters)
