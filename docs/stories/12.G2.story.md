# Story 12.G2 — Critique Verdict Gate Hook

## Description

Spec Pipeline Phase 5 (Critique), defined in
`.aiox-core/development/tasks/spec-critique.md`, produces a `critique.json`
file containing a `verdict` field with one of three values: `APPROVED`,
`NEEDS_REVISION`, or `BLOCKED`. The task spec marks this gate as
`verification: { type: gate, blocking: true, verdict_field: verdict }` and the
documented pipeline handoff says Phase 6 (Plan, owned by @architect) should
only proceed `condition: Critique verdict is APPROVED`.

However, this blocking behaviour is currently **only descriptive** (documented
in the task markdown) — there is no automated hook that actually prevents a
user or agent from invoking the Phase 6 Plan command before `critique.json`
exists, or while its verdict is anything other than `APPROVED`. This story
closes that gap with a deterministic `PreToolUse` hook.

## Acceptance Criteria

1. [ ] AC1: New hook file `.claude/hooks/enforce-spec-critique-gate.cjs` created.
2. [ ] AC2: Matcher = `Bash` commands matching the Phase 6 (Plan) trigger pattern — e.g. `aiox *plan`, `aiox *execute-spec-plan`, or the actual CLI invocation used to start the Plan phase (verify exact command string against `.aiox-core/development/tasks/plan-create-implementation.md` and `.aiox-core/development/workflows/spec-pipeline.yaml` before finalizing the matcher regex — do not assume).
3. [ ] AC3: Hook verifies whether `critique.json` exists in the same spec directory as the story being planned (e.g. `docs/stories/{storyId}/spec/critique.json`, matching the path convention used by `requirements.json`/`complexity.json` in `spec-critique.md`'s `inputs`/`outputs` definitions).
4. [ ] AC4: If `critique.json` does not exist → BLOCK with message: `"Critique not run. Execute Phase 5 first."`
5. [ ] AC5: If `critique.json` exists but its `verdict` field is not `"APPROVED"` → BLOCK with message: `"Critique verdict is {verdict}. Must revise before proceeding."` (interpolate the actual verdict value: `NEEDS_REVISION` or `BLOCKED`).
6. [ ] AC6: If `critique.json` exists and `verdict === "APPROVED"` → ALLOW, and log the decision (allow) via the existing gate-logger JSONL convention.
7. [ ] AC7: Hook registered in `.claude/settings.json` under the `PreToolUse` `Bash` matcher block.
8. [ ] AC8: Tests added to `tests/hooks/enforcement.test.js` covering 3 cases: (a) missing critique.json → block, (b) critique.json present with verdict NEEDS_REVISION or BLOCKED → block, (c) critique.json present with verdict APPROVED → allow.
9. [ ] AC9: Metrics recorded in `.synapse/metrics/hook-metrics.json` under a new key `spec-critique-verdict-gate: { gatesEnforced, allowed, blocked }`.

### Graceful degradation

Per `.claude/rules/enforcement-gates.md`, enforcement tooling must never block
development by *failing* (as opposed to correctly blocking a violation). If
the hook cannot determine the storyId/spec directory from the Bash command
(e.g., command does not specify a story ID), the hook should warn-and-proceed
rather than hard-block — log this as a `warn` decision, not a `block`. Only
hard-block when the spec directory IS resolvable and critique.json is either
missing or not APPROVED.
