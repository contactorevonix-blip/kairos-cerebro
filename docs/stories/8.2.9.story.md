# Story 8.2.9: CI/CD Integration

**Status:** Ready
**Story ID:** 8.2.9
**Created:** 2026-06-11
**Epic:** EPIC-8 — Framework Evolution (Phase 2: IDS Enhancement)
**PRD Source:** `docs/prd/epic-8/phase-2-ids.md#829-cicd-integration-2sp`
**Story Points:** 2

---

## Story

**As a** framework maintainer,
**I want** the G6 registry-integrity gate to run in GitHub Actions on every PR (in addition to the local pre-push hook from 8.2.1), with PR comments explaining required fixes,
**so that** registry integrity is enforced even for contributors who bypass or don't have the local `.husky/pre-push` hook configured, and PR reviewers get actionable feedback automatically.

---

## Acceptance Criteria

1. GitHub Action: `.github/workflows/ids-g6-gate.yml`
   - Runs on PR: Check registry integrity
   - Blocks merge if unregistered entities added
   - Comments on PR with required fixes
2. Pre-push hook: `.husky/pre-push` calls G6 gate
3. Fast-fail: G6 <10s (doesn't slow down dev loop)
4. Logs: All G6 decisions logged to `.aiox/gate-logs/art-ids-g6-*.jsonl`

## Risks

- **Risk:** AC2 ("Pre-push hook: `.husky/pre-push` calls G6 gate") appears to duplicate Story 8.2.1 AC1 ("Git hook: `.husky/pre-push` runs G6 gate"). If 8.2.1 already wires `.husky/pre-push`, this AC may already be satisfied by the time 8.2.9 starts.
- **Mitigation:** When picking up this story, @dev should first verify whether `.husky/pre-push` already invokes the G6 gate (from 8.2.1's Task 3). If so, AC2 is already satisfied — mark it complete with a note pointing to 8.2.1's implementation, do not re-implement. This story's NEW work is primarily AC1 (GitHub Actions workflow) and AC4 confirmation in the CI context.

- **Risk:** AC4 ("Logs: All G6 decisions logged to `.aiox/gate-logs/art-ids-g6-*.jsonl`") — `.aiox/` is typically runtime/gitignored (per `.claude/rules/agent-handoff.md`: "Storage... `.aiox/handoffs/` (runtime, gitignored)"). In a GitHub Actions runner, `.aiox/gate-logs/` would be ephemeral (lost when the runner shuts down) unless explicitly uploaded as an artifact or persisted elsewhere.
- **Mitigation:** For the CI context (AC1), G6 decision logs should either be (a) uploaded as a GitHub Actions artifact for the PR run, (b) posted directly in the PR comment (AC1's "Comments on PR with required fixes" already covers the human-readable side), or (c) both. The local `.husky/pre-push` context (AC2/8.2.1) can continue writing to `.aiox/gate-logs/` as-is since that's a local developer machine. Document the chosen approach in Dev Agent Record.
