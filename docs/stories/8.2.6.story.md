# Story 8.2.6: Change Log Automation

**Status:** Ready
**Story ID:** 8.2.6
**Created:** 2026-06-11
**Epic:** EPIC-8 — Framework Evolution (Phase 2: IDS Enhancement)
**PRD Source:** `docs/prd/epic-8/phase-2-ids.md#826-change-log-automation-15sp`
**Story Points:** 1.5

---

## Story

**As a** framework maintainer investigating why an entity changed,
**I want** every entity registry modification (created, modified, deleted, adapted) automatically logged to an append-only changelog with before/after metadata,
**so that** the IDS has an auditable history of how the framework evolved, without relying on git log archaeology across hundreds of commits.

---

## Acceptance Criteria

1. File: `.aiox-core/data/entity-changelog.jsonl` (append-only log)
2. Entry format:
   ```json
   {
     "timestamp": "2026-06-11T...",
     "entity_id": "story-8.2.6",
     "action": "created|modified|deleted|adapted",
     "agent": "@dev",
     "commit": "abc123...",
     "before": {...old metadata},
     "after": {...new metadata}
   }
   ```
3. Auto-capture: On every registry update
4. Query API: `aiox ids changelog {entity-id}` (show entity history)
5. Retention: Keep full history (no purge)

## Risks

- **Risk:** "Auto-capture: On every registry update" (AC3) requires hooking into ALL code paths that write to `entity-registry.yaml` — by this point in Phase 2, that includes 8.2.2's auto-heal, 8.2.5's auto-registration, and any pre-existing manual sync script (`populate-entity-registry.js`).
- **Mitigation:** Centralize registry writes through a single write function/module if one doesn't already exist (REUSE>ADAPT>CREATE per Article IV-A — check 8.2.2's `registry-auto-heal.js` for a reusable write path first), and have the changelog hook live at that single point rather than being duplicated across 8.2.2/8.2.5/legacy script call sites.

- **Risk:** AC2's example entry includes a `"commit": "abc123..."` field — capturing the current git commit hash at the moment of a registry write requires shelling out to git or reading `.git/HEAD`, which may not always be available/clean (e.g., mid-rebase, detached HEAD).
- **Mitigation:** Treat `commit` as best-effort/nullable — if the git commit hash cannot be reliably determined at write time, log `null` rather than failing the registry update. Document this in Dev Notes for @qa.

### Architecture Notes

- No specific guidance found in `docs/architecture/` for changelog persistence. The `.aiox/gate-logs/` and (forthcoming) `.aiox/ids-heal-log.jsonl` patterns are the closest precedents and should be followed for consistency, even though `entity-changelog.jsonl` lives in `.aiox-core/data/` (tracked/committed per `boundary.exceptions` in `core-config.yaml`: `.aiox-core/data/**` is an exception to framework protection) rather than `.aiox/` (runtime/gitignored). This distinction matters: `entity-changelog.jsonl` being in `.aiox-core/data/` implies it IS git-tracked, unlike the gate-logs/heal-log. @dev should confirm this is intentional (audit trail should be version-controlled) with @architect.

### Testing

- Format test: every appended line must be valid JSON and conform to the AC2 schema (use a JSON Schema if helpful, similar to 8.1.1's pattern).
- Coverage test: trigger a registry change via 8.2.2's auto-heal (e.g., dry-run → apply) and confirm a corresponding changelog entry is created with correct `action: "modified"` (or `"created"`/`"deleted"` as appropriate) and accurate `before`/`after`.
- Append-only test: confirm that running multiple registry updates results in a strictly growing file (line count only increases, never decreases or rewrites prior lines).
- Query test: `aiox ids changelog {entity-id}` for an entity with multiple historical entries returns them in chronological order.
- Graceful degradation test: simulate an environment where `git rev-parse HEAD` fails (e.g., outside a git repo) — confirm the registry write still succeeds with `commit: null`.
