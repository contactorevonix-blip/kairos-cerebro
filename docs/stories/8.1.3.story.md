# Story 8.1.3: Gate Metrics Integration

**Status:** Ready
**Story ID:** 8.1.3
**Created:** 2026-06-11
**Epic:** EPIC-8 — Framework Evolution (Phase 1: Observability & Metrics)
**PRD Source:** `docs/prd/epic-8/phase-1-observability.md#813-gate-metrics-integration-2sp`
**Story Points:** 2

---

## Story

**As a** framework maintainer,
**I want** constitutional enforcement gate logs to be parsed and aggregated into queryable metrics,
**so that** `aiox metrics` can report violations/day, blocks/week, and override frequency with full audit traceability back to the original gate-log entry.

---

## Acceptance Criteria

1. Task: `parse-gate-logs.js` reads `.aiox/gate-logs/` daily files
2. Metrics extracted: article + decision + reason + agent + timestamp
3. Aggregations: violations/day, blocks/week, override frequency
4. Performance: Parse 30 days of logs <2s
5. Audit trail: All metrics include original gate-log entry reference

**Example Metric:**
```json
{
  "metric_id": "gate-block-count",
  "timestamp": "2026-06-11T23:59:59Z",
  "article": "art-ii-agent-authority",
  "decision": "block",
  "count": 3,
  "period": "day"
}
```

## Tasks / Subtasks

- [ ] Task 1 (AC: 1, 2) — Implement `parse-gate-logs.js`
  - [ ] Read all `.aiox/gate-logs/{article}-{YYYY-MM-DD}.jsonl` files within the requested date range
  - [ ] Parse each JSON line into a normalized record: `{ article, gate, decision, reason, agent, operation, timestamp, sourceFile, lineNumber }`
  - [ ] Skip and log (do not crash on) malformed lines or missing files
- [ ] Task 2 (AC: 3) — Implement aggregations
  - [ ] `violations/day` — count of `decision: "block"` + `decision: "violation"`-equivalent entries grouped by day
  - [ ] `blocks/week` — count of `decision: "block"` grouped by ISO week
  - [ ] `override frequency` — count of `decision: "override"` per article, per period
  - [ ] Output aggregations in the metric shape defined by 8.1.1 schema (e.g., the `gate-block-count` example in the PRD)
- [ ] Task 3 (AC: 4) — Performance optimization
  - [ ] Benchmark parsing 30 days of gate-logs, target <2s
  - [ ] If needed, add an incremental/cached parse mode (only parse new lines since last run) — document the approach in Dev Notes
- [ ] Task 4 (AC: 5) — Audit trail
  - [ ] Ensure every aggregated metric record includes a reference to its source gate-log entry/entries (file path + line number, or a stable id)
- [ ] Task 5 — Wire into `aiox metrics` CLI (from 8.1.2)
  - [ ] `aiox metrics list` should now show gate-derived metric types (e.g., `gate-block-count`, `gate-override-frequency`)
  - [ ] `aiox metrics get gate-block-count` returns the aggregation + history

### Gate Log Format (from `.claude/rules/enforcement-gates.md`)

Every decision is appended to `.aiox/gate-logs/{article}-{YYYY-MM-DD}.jsonl`, one JSON object per line:

```json
{"timestamp":"2026-06-08T19:00:00.000Z","article":"art-ii-agent-authority","gate":"agent-authority","decision":"block","reason":"git push is exclusive to @devops ...","agent":"@dev","operation":"git push"}
```

`decision` is one of: `allow`, `block`, `override`, `warn`, `warn-and-proceed`.

### Gate Inventory (for reference — which articles produce gate-logs)

| Article | Gate | Severity |
|---------|------|----------|
| II — Agent Authority | agent-authority | BLOCK |
| III — Story-Driven | story-driven | BLOCK |
| IV — No Invention | no-invention | BLOCK |
| V — Quality First | quality-gates | BLOCK |
| VI-VII — Framework Boundary | quality-gates | BLOCK |
