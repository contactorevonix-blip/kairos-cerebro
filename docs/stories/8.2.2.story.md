# Story 8.2.2: Registry Auto-Heal

**Status:** Ready
**Story ID:** 8.2.2
**Created:** 2026-06-11
**Epic:** EPIC-8 — Framework Evolution (Phase 2: IDS Enhancement)
**PRD Source:** `docs/prd/epic-8/phase-2-ids.md#822-registry-auto-heal-2sp`
**Story Points:** 2

## Story

**As a** framework maintainer,
**I want** the entity registry to detect and repair common drifts (missing entries, deleted-but-listed entities, stale metadata, circular dependencies) automatically or via a guided dry-run,
**so that** the IDS registry stays accurate without requiring developers to remember to run `populate-entity-registry.js` manually after every change.

---

## Acceptance Criteria

1. Task: `.aiox-core/core/ids/registry-auto-heal.js` (healing logic)
2. Detection patterns (auto-triggered when):
   - Entity file exists in filesystem but missing from registry
   - Entity listed in registry but file deleted
   - Entity has stale timestamps (older than file mtime)
   - Circular dependency detected in entity graph
3. Repair actions:
   - Auto-add missing entities (populate metadata from file headers)
   - Auto-remove deleted entities (log as audit trail)
   - Fix stale metadata (update timestamps)
   - Suggest cycle resolution (not auto-fix, user decision)
4. Dry-run mode: `aiox ids heal --dry-run` (show changes without applying)
5. Logging: All heals logged to `.aiox/ids-heal-log.jsonl`

## Risks

- **Risk:** Auto-heal could mass-modify `.aiox-core/data/entity-registry.yaml` (823 entities) incorrectly, corrupting the IDS source of truth.
- **Mitigation:** AC4 dry-run mode is mandatory and must be the default/recommended first step; AC5 logging to `.aiox/ids-heal-log.jsonl` provides full audit trail; per PRD "Safety: All heals must be reversible (git-tracked)" — every applied heal must result in a normal git diff that can be reverted.

- **Risk:** Cycle detection (AC2) may find existing cycles already present in the live 823-entity registry, producing noisy first-run output.
- **Mitigation:** AC3 explicitly defers cycle resolution to the user ("Suggest cycle resolution (not auto-fix, user decision)") — first run should report findings without blocking.

## Tasks / Subtasks

- [ ] Task 1 (AC: 1) — Create `.aiox-core/core/ids/registry-auto-heal.js`
  - [ ] Load `.aiox-core/data/entity-registry.yaml` (823 entities, `checksumAlgorithm: sha256`)
  - [ ] Build an in-memory entity graph using `usedBy[]` and `dependencies[]` for traversal
- [ ] Task 2 (AC: 2) — Implement detection patterns
  - [ ] Pattern A: scan filesystem for files under registered paths/layers not present as `path:` in any registry entity
  - [ ] Pattern B: for each registry entity, verify `path:` still exists on disk; flag if deleted
  - [ ] Pattern C: compare `lastVerified` timestamp against file `mtime`; flag entities where `mtime > lastVerified`
  - [ ] Pattern D: reuse/extend the cycle-detection logic from Story 8.2.1's G6 gate (check 2) — do not reimplement if 8.2.1 already produced a reusable cycle-detection function
- [ ] Task 3 (AC: 3) — Implement repair actions
  - [ ] Auto-add: for Pattern A findings, populate a new registry entry — `path`, `type`, inferred `purpose` (from file header/first heading), `keywords` (from filename tokens), default `adaptability.score`, `checksum` (sha256 per existing `checksumAlgorithm`), `lastVerified: now`
  - [ ] Auto-remove: for Pattern B findings, remove the entity entry and log the removed entity's full prior state to `.aiox/ids-heal-log.jsonl`
  - [ ] Fix stale metadata: for Pattern C findings, recompute `checksum` and update `lastVerified`
  - [ ] Cycle resolution: for Pattern D findings, output a human-readable cycle report (entity chain) — do NOT auto-modify `dependencies[]`/`usedBy[]`
- [ ] Task 4 (AC: 4) — Dry-run mode
  - [ ] Implement `aiox ids heal --dry-run` — runs all detection + repair logic but writes a diff/report instead of modifying `entity-registry.yaml`
  - [ ] Implement `aiox ids heal` (no flag) — applies repairs and writes the updated registry file
- [ ] Task 5 (AC: 5) — Logging
  - [ ] For every repair action (applied or dry-run-proposed), append a JSON line to `.aiox/ids-heal-log.jsonl` with at minimum: `timestamp`, `action` (add|remove|fix-metadata|cycle-detected), `entity_id`, `before`, `after`, `mode` (dry-run|applied)

The research doc (`ids-enhancement-gaps.md#gap-2`) proposes two heal levels: "Level 1: Lazy Sync (WARN only)" and "Level 2: Full Sync (BLOCK, requires override)". ### Testing

- Dry-run test: run `aiox ids heal --dry-run` against the current registry; assert `entity-registry.yaml` is byte-identical before/after.
- Detection test: temporarily create an unregistered file under a tracked layer, run dry-run, assert it appears in the proposed-additions report; remove the file, re-run, assert no residual entries.
- Repair test: apply a heal that fixes a stale-timestamp entity; assert `.aiox/ids-heal-log.jsonl` records `before`/`after` with the old and new `lastVerified`/`checksum`.
- Idempotency test: run `aiox ids heal` twice consecutively; the second run should report zero changes.
