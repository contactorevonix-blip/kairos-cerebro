## Details

### Testing Approach

**MAESTRO Layer Validation:**
- **L1 (Core):** QA gate definition, verdict logic
- **L2 (Templates):** QA checklist template applies correctly
- **L3 (Config):** QA rules (story-lifecycle.md) load and apply
- **L4 (Runtime):** @qa executes in real story workflow (paired with @dev)

**Task-First Contract:**
```
Task: Test @qa Agent
  Inputs: [agent config, story with ACs, mock test results]
  Outputs: [gate verdict, issues list, metrics log]
  Success: All 8 ACs passing, verdicts consistent
```

### 7-Point Quality Checks

Each @qa gate verifies:
1. Code review (patterns, readability, maintainability)
2. Unit tests (adequate coverage, all passing)
3. Acceptance criteria (all met per story AC)
4. No regressions (existing functionality preserved)
5. Performance (within acceptable limits)
6. Security (OWASP basics verified)
7. Documentation (updated if necessary)

### Research Alignment

**Ensemble Validation** (research rec, EPIC-12 §5.5):
- @qa is the second validation role after @dev
- QA verdict is binding (determines story promotion)
- QA evidence is auditable (logged for compliance)

**Specification-Driven Determinism:**
- Gate criteria are precise (not subjective)
- Same story → same verdict, regardless of QA agent instance
