## Acceptance Criteria

- [ ] **AC1 (TEST-001 — HIGH, BLOCKING):** Fix `npm test` so it exits 0 (currently exits 1: 71 pass / 0 fail / 9 cancelled in `tests/hooks/auto-contextualization-hook.test.js`). Root cause: `agent-activation-tracker.cjs:204` calls `timer.unref()` on the 2000ms guard timer inside `runContextPhasesGuarded`; when the HANGING_ENGINE test awaits the timeout budget, the event loop has no live handle, so `node --test` reports "Promise resolution is still pending but the event loop has already resolved" and cancels that test plus 8 cascading tests.
  - **Suggested action (from gate):** "Return @dev to fix: in the test, keep the loop alive during the timeout wait (e.g. a `ref()`'d guard timer in the test, or fake timers); OR make the guard timer `ref()`'d while awaiting and `unref()` only on the production hook path. Re-run `npm test` to exit 0 before declaring the epic Done."
  - **Done when:** `npm test` exits 0 with 0 cancelled, 0 failed (71+ pass).

- [ ] **AC2 (REQ-001 — MEDIUM):** Resolve the discrepancy between Story 5.3.1 AC1 ("All 10 phases implemented and executable via `aiox run-context-engine`") and reality (no such CLI exists in `bin/` or `.aiox-core/`).
  - **Suggested action (from gate):** "Either add the CLI binding (route via @dev) or have @po amend AC1 to reflect the class-only delivery; record the decision in the Change Log."
  - **Done when:** Either (a) `aiox run-context-engine` exists and is executable and invokes `.synapse/context-engine/engine.js`, OR (b) @po has amended Story 5.3.1 AC1 wording to reflect the class-only delivery, with the chosen option recorded in Story 5.3.1's Change Log.

- [ ] **AC3 (REL-001 — MEDIUM):** Wire `.synapse/context-engine/engine.js` to the registry module delivered by Story 5.3.3 (`.synapse/context-registry.js`), reconciling the path/format mismatch between the engine's inline `loadRegistry`/`saveRegistry` (which `JSON.parse()`s a path named `.synapse/context-registry.yaml`) and the two coexisting registry files (`.synapse/context-registry.json` and `.synapse/context-registry.yaml`).
  - **Suggested action (from gate):** "Wire `engine.js` to `require('.synapse/context-registry.js')` (or reconcile path/format so engine + module use one store). Track as a follow-up story; current engine self-test masks the gap."
  - **Done when:** `engine.js` Phase 5/Phase 10 registry reads/writes go through `.synapse/context-registry.js`'s exported API (or a single reconciled store/format is used by both); `tests/context-registry/registry.test.js` (13/13) and `tests/auto-contextualization/engine.test.js` (41/41 incl. phases) both remain green after the change.

- [ ] **AC4 (MNT-001 — LOW):** Reconcile Story 5.3.1's Tasks 7-11 (Phases 6-10) checkbox state and Dev Agent Record "Remaining Work" note with the actual implementation — Phases 6-10 already exist inline in `engine.js`.
  - **Suggested action (from gate):** "Reconcile task checkboxes and Dev Record with the inline implementation, or split out modules to match the stated File List."
  - **Done when:** Story 5.3.1 Tasks 7-11 checkboxes accurately reflect implementation state (checked if Phases 6-10 are confirmed inline-implemented and tested), and the Dev Agent Record / File List no longer lists Phases 6-10 as "Remaining Work" without justification. Change documented in Story 5.3.1's Change Log.

  - **Suggested action (from gate):** "Enforce qa-gate.md 'Post-Gate Status Update (MANDATORY)' — update the header Status field + append a Change Log gate entry in the same step the verdict is decided."
  - **Note:** `qa-gate.md` lives at `.aiox-core/development/tasks/qa-gate.md`, which is L2 (Framework Templates, NEVER modify per `.claude/CLAUDE.md` boundary table — extend-only). This AC is therefore scoped to L4 project runtime: reinforce the existing rule in `.claude/rules/story-lifecycle.md` (which already documents the `InReview → Done` transition as @qa's responsibility, "MUST update story status and Change Log before reporting the gate result") with an explicit, repeatable checklist step covering: (1) update header `Status:` field, (2) append Change Log row with gate verdict and date, (3) sync any redundant/duplicate Status lines in the story footer — performed atomically.
