## Story
**As a** founder do Kairos Check,
**I want** que a landing page nunca apresente números fabricados como se fossem actividade real,
**so that** developers técnicos que inspeccionem o código, a API ou o feed de actividade não percam confiança no produto — e o produto cumpra a Regra de Ouro #1 do projecto ("Nunca inventar dados — só dados reais").

## Contexto / Por que agora

Esta story resolve o débito **UX-012** identificado na Brownfield Discovery (Fase 10), e está classificada **P1** no roadmap de remediação (`EPIC-001-tech-debt-remediation.md`). É a próxima story de valor após o fecho da PHASE 4 (enforcement constitucional do framework AIOX) — desta vez aplicando o mesmo princípio de "No Invention" (Art. IV da Constitution) ao **produto real**, não ao framework.

**Achado crítico (verificado nesta refinação — vai além do diagnóstico original):**
A label da activity feed na landing page diz literalmente **"Real fraud detections — powered by Kairos Check"**, mas o número ao lado (`#activity-count`) é alimentado por:
1. Um valor inicial fabricado, calculado server-side por `counterBase()` — uma função de `180 + dias_desde_lançamento × 400 + variação_horária`;
2. Um auto-incremento client-side (`count += 1` a cada 11 entradas simuladas, ~a cada 45s) que **nada tem a ver com tráfego real**.

Ou seja: não é só o endpoint `/api/stats/counter` que inventa números — o próprio runtime do browser continua a inflar o contador depois da página carregar, por baixo de uma etiqueta que afirma serem dados "Real". Isto é uma violação directa e visível do Artigo IV (No Invention) da Constitution, em produção, num elemento com `aria-live="polite"` (acessível a leitores de ecrã como informação factual).

## Acceptance Criteria

- [ ] **AC1:** Endpoint `/api/stats/counter` usa exclusivamente dados reais (`verifyRequests`) — nunca `counterBase()`.
- [ ] **AC2:** Landing page mostra copy honesta de early-stage quando `verifyRequests === 0`.
- [ ] **AC3:** `counterBase()` é removido de `landing-page.js` e `server.js`.
- [ ] **AC4:** Auto-incremento artificial client-side é removido (sem `count += 1` por timer).
- [ ] **AC5:** Label "Real fraud detections" é consistente com dados exibidos.
- [ ] **AC6:** Proof bar comportamento permanece inalterado (sem regressão).
- [ ] **AC7:** API `/api/stats/counter` retorna sempre dados reais com `cache-control: no-store`.
- [ ] **AC8:** Landing page renderiza sem erros, dados coerentes com `metrics.json`.

## AUTO-DECISION (River, @sm)

**[AUTO-DECISION]** Implementar a **Opção A** (contador oculto / copy honesta de early-stage) em vez da Opção B (mostrar número real-ou-zero) → **(reason:** a Opção B continua a expor `verifyRequests` real junto a uma label que diz "Real fraud detections" — o que é tecnicamente correcto — MAS a investigação desta refinação revelou que o problema vai além do endpoint: existe um **auto-incremento client-side activo** (`count += 1` a cada ~45s) que continuaria a mentir mesmo com o endpoint corrigido, a menos que também seja removido. Dado que (a) o produto está em fase beta com tráfego real provavelmente baixo/zero, (b) mostrar "0 today" ou um número que só sobe por timer local destrói mais confiança do que não mostrar nada, e (c) a Regra de Ouro #1 do Pedro é explícita ("Nunca inventar dados — só dados reais ou secções honestas"), a opção mais segura e alinhada com a filosofia já aplicada à proof bar (`showProofBar = rawRequests > 0`, oculta quando não há dados) é: ocultar o contador dinâmico e substituir por copy honesta tipo "Acesso antecipado — os primeiros checks aparecem aqui em tempo real" SEM número, ATÉ que `verifyRequests > 0` de forma sustentada — ponto em que pode evoluir naturalmente para Opção B numa story futura. Isto resolve TODOS os ACs (1-8) com uma mudança coerente, sem deixar um "meio-termo" fabricado.**)
