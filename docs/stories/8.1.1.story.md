# Story 8.1.1: Metrics Schema Design

**Status:** Ready
**Story ID:** 8.1.1
**Created:** 2026-06-11
**Epic:** EPIC-8 — Framework Evolution (Phase 1: Observability & Metrics)
**PRD Source:** `docs/prd/epic-8/phase-1-observability.md#811-metrics-schema-design-15sp`
**Story Points:** 1.5

---

## Story

**As a** framework maintainer,
**I want** a unified 3-layer metrics data model (Gate, Session, Framework) defined as documentation + JSON schema + DB schema,
**so that** all future observability work (CLI collector, dashboard, persistence) builds on a single, extensible, non-invented data model.

---

## Acceptance Criteria

- [ ] **AC1:** Schema document `docs/architecture/metrics-schema.md` created with Gate-layer, Session-layer, and Framework-layer metrics documented
- [ ] **AC2:** JSON schema file `.aiox-core/data/metrics-schema.json` created and validated against metrics data
- [ ] **AC3:** Database schema `migrations/001-metrics-init.sql` created (ready for Phase 1.7 finalization)
- [ ] **AC4:** Traceability matrix confirms no invented metrics; all metrics sourced from existing data (gate-logs, hook-metrics, task-logs)

---

## Dependencies

- **Depends on:** EPIC-7 complete (enforcement gates active) — ✅ satisfied
- **Blocks:** 8.1.2 (CLI Metrics Collector), 8.1.3 (Gate Metrics Integration), 8.1.7 (Metrics Persistence)

---

## Risks

- **Risk:** Schema too narrow (can't capture future metrics)
- **Mitigation:** Design as extensible (add new metric types without schema migration) — use a generic `metric_type` string key + JSONB payload pattern rather than rigid columns per metric.

## Tasks / Subtasks

- [ ] Task 1 (AC: 1) — Author `docs/architecture/metrics-schema.md`
  - [ ] Document Gate-layer metrics: source `.aiox/gate-logs/*.jsonl` — fields: article, decision, reason, agent, operation, timestamp
  - [ ] Document Session-layer metrics: source `.synapse/metrics/hook-metrics.json` — fields: session id, active_agent, prompt_count, story progress, quality gate results
  - [ ] Document Framework-layer metrics: trend aggregations derived from Gate + Session layers (e.g., violations/day, story velocity, gate compliance %)
  - [ ] Include a "no invented metrics" traceability table mapping each documented metric to its source file/field
- [ ] Task 2 (AC: 2) — Create `.aiox-core/data/metrics-schema.json`
  - [ ] Define JSON Schema (draft-07 or 2020-12) describing the 3 layers as typed objects
  - [ ] Use a generic `metric_type` (string) + `value`/`payload` pattern for extensibility
  - [ ] Validate the JSON schema file is syntactically valid (e.g., `node -e "JSON.parse(require('fs').readFileSync('.aiox-core/data/metrics-schema.json','utf8'))"` plus a JSON-Schema validator if available)
- [ ] Task 3 (AC: 3) — Create `migrations/001-metrics-init.sql`
  - [ ] Define `metrics` table (id, metric_type, key, value, source, agent, article, timestamp)
  - [ ] Define `metric_history` table (aggregated rollups: period, metric_type, aggregate_value, computed_at)
  - [ ] Add indexes on `timestamp`, `article`, `agent` for range queries (prepares for 8.1.7 performance requirement)
  - [ ] Document this migration is consumed by Story 8.1.7 (do not implement DB connection here — schema only)
- [ ] Task 4 (AC: 4) — Traceability validation
  - [ ] Cross-check every metric type defined in `metrics-schema.md` / `.json` / `.sql` against an existing source: `.aiox/gate-logs/`, `.synapse/metrics/hook-metrics.json`, `.aiox/task-logs/`
  - [ ] Flag and remove any metric without a traceable source
