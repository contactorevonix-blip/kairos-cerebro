## Summary

Implement task auto-discovery and auto-activation based on SYNAPSE context. Enable 200+ AIOX tasks in `.aiox-core/development/tasks/` to be automatically invoked based on story type, layer context, and workflow state. Operationalize the Task-First principle: no code workflow begins without a task being activated.

**Why:** AIOX has 200+ tasks defined, but system never auto-activates them. Every story requires manual navigation to find + invoke the right task. This story creates an auto-activation engine that detects context and surfaces relevant tasks.

---

## Acceptance Criteria

- [ ] AC1: Task auto-discovery working
  - Scan `.aiox-core/development/tasks/` (all task .md files)
  - Extract task metadata (category, trigger_context, dependencies)
  - Build task registry (cached, updated on story changes)
  - 200+ tasks indexed and available

- [ ] AC2: Task auto-activation based on story type
  - Detect story type from title/description/AC (feature, bug, refactor, etc.)
  - Match against task categories (create-feature-task, fix-bug-task, etc.)
  - Auto-suggest 3-5 relevant tasks to developer
  - Surface via SYNAPSE layer or CLI suggestion

- [ ] AC3: Context-aware task activation
  - Use L3 Workflow Layer to detect active workflow (SDC, QA Loop, Spec Pipeline, Brownfield)
  - Activate workflow-specific tasks automatically
  - Example: Starting SDC story auto-activates (create-story, dev-develop-story, qa-gate, devops-push)

- [ ] AC4: Task workflow execution automated
  - Tasks with `automated: true` execute automatically (no manual invocation)
  - Tasks with `interactive: false` run silent (no prompts)
  - Task prerequisites verified before execution
  - Task outputs captured for metrics + audit

- [ ] AC5: Task-first principle operationalized
  - Story template includes "Suggested Tasks" section
  - Tasks auto-injected based on story type
  - Developer cannot skip task activation (soft enforcement)
  - Metrics track: tasks_available, tasks_activated, tasks_completed

- [ ] AC6: Metrics + debugging
  - `.synapse/metrics/hook-metrics.json` tracks:
    - `tasksIndexed`: count of indexed tasks
    - `tasksActivated`: count of activated tasks (per session)
    - `taskExecutionTime`: average execution time
    - `taskAutoActivationRate`: % of tasks auto-activated vs manual

## Implementation Notes

- **Task Registry:** Built from task frontmatter metadata (category, triggers, dependencies)
- **Auto-Activation:** PreToolUse hook detects story type, queries registry, surfaces tasks
- **Graceful Fallback:** If auto-discovery fails, system operates normally (no blocking)
- **Context Leakage:** No task execution without explicit user confirmation (safety)
- **Task Isolation:** Each task runs in sandbox (no cross-task state pollution)

## Tasks / Subtasks

1. **Build task registry** (AC1)
   - Scan `.aiox-core/development/tasks/` for all .md files
   - Extract metadata: `task:`, `category:`, `trigger_context:`, `dependencies:`
   - Create `TASK-REGISTRY.json` with indexed tasks
   - Cache registry, invalidate on file changes

2. **Implement auto-discovery engine** (AC1-AC2)
   - Create `.aiox-core/core/task-discovery/discover-tasks.js`
   - Parse story type from title/AC/description
   - Query registry for matching tasks
   - Return ranked list (relevance score)

3. **Wire up SYNAPSE layer** (AC2-AC3)
   - Update L3 Workflow Layer to return available tasks
   - Use layer output in story template "Suggested Tasks" section
   - Link tasks to workflow phases (SDC: create, dev, qa, push)

4. **Implement task auto-activation** (AC3-AC4)
   - Create PreToolUse hook: detect story start, auto-suggest tasks
   - User selects task (or accepts suggestion)
   - Task executes with auto-activation flag
   - Capture outputs for metrics

5. **Add metrics + reporting** (AC5-AC6)
   - Update hook-metrics.json schema with task counters
   - Log task activations to `.aiox/task-logs/{story-id}.json`
   - Create CLI command: `aiox tasks --list-available` (show registry)
   - Create CLI command: `aiox tasks --executed` (show completed in session)

6. **Test automation scenarios** (all ACs)
   - Test: Story type detection (feature → suggests create-feature-task)
   - Test: Workflow detection (SDC → suggests all 4 phase tasks)
   - Test: Task registration + metadata extraction
   - Test: Task execution in isolation
   - Test: Metrics aggregation

## Dev Notes

**Task Metadata Format (from task .md frontmatter):**
```yaml
task: taskName()
category: [feature|bug|refactor|arch|config|docs]
trigger_context: [story-start|code-review|pre-commit|workflow-phase]
automated: true|false
interactive: true|false
dependencies: [list of prerequisite tasks]
```

**Story Type Detection:**
- Keywords: feature/add/implement → feature
- Keywords: fix/bug/patch → bug
- Keywords: refactor/cleanup/optimize → refactor
- Keywords: arch/design/pattern → arch

**Auto-Activation Safety:**
- Tasks require explicit user confirmation (unless `automated: true` AND `interactive: false`)
- Task sandbox prevents side effects
- Failed tasks are logged but don't block workflow
