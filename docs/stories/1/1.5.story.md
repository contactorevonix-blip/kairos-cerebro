## Summary

Build Fastify REST API with 4 read-only endpoints: `GET /sessions`, `GET /agents`, `GET /tasks`, `GET /hooks`. Each endpoint returns current state snapshot from ParallelMonitorConnector (Story 1.1). Serve as HTTP complement to WebSocket real-time updates (Story 1.2). API enforces Bearer token auth (localhost Bearer token from `.clauderc`).

**Why:** CLI tools and dashboards need both polling (HTTP GET) and streaming (WebSocket) interfaces. HTTP is simpler for CLI scripts, WebSocket is efficient for dashboards.

---

## Acceptance Criteria

- [ ] Create `src/api/Monitor.ts` (Fastify server):
  - Binds to `127.0.0.1:8080` (localhost only, NFR4)
  - Requires Bearer token header: `Authorization: Bearer <token>`
  - Token from `.clauderc` config (Story 4.2 will define this)

- [ ] Endpoints:
  ```
  GET /sessions
    → returns: { sessions: SessionSnapshot[] }
    → latency target: <50ms (NFR1)
    
  GET /agents
    → returns: { agents: { id: string; status: 'idle' | 'active' }[] }
    → latency target: <50ms
    
  GET /tasks
    → returns: { tasks: { id: string; workflow: string; status: string }[] }
    → latency target: <100ms
    
  GET /hooks
    → returns: { hooks: { name: string; last_run: ISO8601; next_run?: ISO8601 }[] }
    → latency target: <100ms
  ```

- [ ] Error responses:
  - `401 Unauthorized` if Bearer token missing/invalid
  - `503 Service Unavailable` if ParallelMonitor unavailable
  - Standard HTTP status codes

- [ ] Monitoring: each endpoint logs to audit (AuditLog from Story 1.3)
- [ ] Health check: `GET /health` returns `{ status: 'ok' }` (no auth required)
- [ ] CORS: disabled (localhost only)
- [ ] Tests: verify latency targets, token validation, response schemas

- **Constitution:** Art. I (CLI first) — API is CLI-compatible
- **Token validation:** implement in Fastify hook, not per-endpoint
