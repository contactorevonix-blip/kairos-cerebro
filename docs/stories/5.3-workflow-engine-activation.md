# Story 5.3 — Workflow Engine Activation

**Epic:** AIOX-OPS (AIOX Operacional 100%) | **Story ID:** 5.3 | **Effort:** 8sp | **Status:** Ready | **Executor:** @architect | **Quality Gate:** @pm

## Story

**As a** AIOX framework operator,
**I want** a documented decision on whether workflow engine flags (`autoClaude.specPipeline/execution/qa`) are enabled or intentionally disabled,
**so that** the configuration is coherent, the rationale is preserved, and the 4 workflows (SDC, QA Loop, Spec Pipeline, Brownfield) operate as designed without ambiguity.

## Goal

Determinar e documentar o estado de `autoClaude.specPipeline.enabled`, `autoClaude.execution.enabled`, e `autoClaude.qa.enabled` em `core-config.yaml` (actualmente todos `false`). Esta é uma decisão arquitectural: devem activar? Se sim, validar que SDC/QA Loop/Spec Pipeline/Brownfield correm ponta-a-ponta. Se não, documentar o porquê explicitamente.

## Contexto / Problema

GAP G4 (EPIC-AIOX-OPERACIONAL.md §3): `core-config.yaml` L362-368 declara `autoClaude.specPipeline.enabled: false`, `execution.enabled: false`, `qa.enabled: false`. Esta configuração não tem documentação associada que explique se é intencional (modo manual seguro) ou se está partido. Sessão após sessão os workflows são invocados manualmente — mas a config afirma que deveriam ser automáticos.

A tensão: "AIOX Operacional" pressupõe zero setup manual, mas as flags `false` podem ser uma decisão de design (evitar automação excessiva) ou um estado incompleto. Só uma decisão arquitectural documentada resolve esta ambiguidade.

## Acceptance Criteria

- [ ] **AC1:** Estado actual das 3 flags verificado e documentado: `autoClaude.specPipeline.enabled`, `autoClaude.execution.enabled`, `autoClaude.qa.enabled`. Para cada uma: valor actual, secção no `core-config.yaml`, e impacto de activar.
  - *Given* `core-config.yaml`, *When* lido, *Then* cada flag tem valor confirmado e descrição de impacto.

- [ ] **AC2:** Decisão arquitectural documentada por flag — para cada uma das 3 flags, decisão explícita: ACTIVATE | KEEP_DISABLED. Razão técnica obrigatória (não apenas "por design").
  - *Given* análise de impacto, *When* decisão é tomada, *Then* existe `docs/architecture/ADR-workflow-engine-flags.md` com a decisão e justificação.

- [ ] **AC3:** Para cada flag marcada ACTIVATE: validar que o workflow correspondente executa ponta-a-ponta sem direcção manual. Produzir evidência (log de execução ou test output).
  - *Given* flag activada, *When* workflow é invocado, *Then* completa sem intervenção humana e sem erro.

- [ ] **AC4:** Para cada flag marcada KEEP_DISABLED: documentar o caso de uso que requer activação futura e as condições de activação segura.
  - *Given* decisão KEEP_DISABLED, *When* documentada, *Then* ADR inclui "Conditions for future activation" com critérios mensuráveis.

- [ ] **AC5:** `core-config.yaml` actualizado para reflectir a decisão: flags correctas para cada workflow, comentários inline explicando o estado. Backward-compatible (sem quebrar hooks existentes).
  - *Given* decisão documentada no ADR, *When* config actualizada, *Then* `npm test` passa sem erros de configuração.

- [ ] **AC6:** Risk assessment completado — para cada flag ACTIVATE, documentar: risco de activação, plano de rollback (`git revert` da mudança em config), e mecanismo de desactivação de emergência.

## Scope

**IN:**
- Análise e decisão das 3 flags: `specPipeline.enabled`, `execution.enabled`, `qa.enabled`
- ADR documentando a decisão
- Actualização de `core-config.yaml` com comentários inline
- Validação e2e dos workflows activados (se aplicável)
- Risk assessment por flag

**OUT:**
- Reescrever o workflow engine em `.aiox-core/core/` (L1 — NEVER modify)
- Modificar templates de workflow em `.aiox-core/development/` (L2 — extend-only)
- Stories downstream (5.4, 5.5, 5.6) que dependem do output desta decisão
- Criar novos workflows (não existe no scope do epic)

## Complexity Estimate

8sp — análise de impacto de 3 flags × 4 workflows + ADR + validação e2e (se flags activadas) + risk assessment. Alta complexidade: é uma decisão arquitectural com implicações no comportamento de todo o sistema. HIGH risk porque activar flags erradas pode partir automações em produção.

## Business Value

Elimina a ambiguidade recorrente no STATE.md sobre "configured vs. operationalized". Com a decisão documentada, qualquer agente que leia a config sabe exactamente o que está activo e porquê — sem ter de inferir da história de sessions anteriores.

## Risks

- **R1 (HIGH):** Activar `execution.enabled: true` sem validação pode disparar automações inesperadas durante sessions normais. Mitigação: testar em ambiente isolado, activar uma flag de cada vez, manter `git revert` como rollback imediato.
- **R2 (MEDIUM):** Workflow e2e tests podem revelar bugs nos engines que não estavam visíveis em modo manual. Mitigação: documentar bugs como GAPs accionáveis sem bloquear a story.
- **R3 (LOW):** ADR pode conflituar com decisões anteriores não documentadas. Mitigação: consultar STATE.md e git log para contexto histórico antes de decidir.

### 4 Workflows em causa (fonte: `.claude/rules/workflow-execution.md`)

| Workflow | Flag | Modo actual |
|----------|------|-------------|
| Story Development Cycle (SDC) | `execution.enabled` | Manual |
| QA Loop | `qa.enabled` | Manual |
| Spec Pipeline | `specPipeline.enabled` | Manual |
| Brownfield Discovery | `execution.enabled` | Manual |

### Localização do ADR

Criar em `docs/architecture/ADR-workflow-engine-flags.md`. Seguir formato ADR standard:
- Status: Accepted
- Context: descrição do problema
- Decision: o que foi decidido por flag
- Consequences: impactos positivos e negativos
- Conditions for activation (se KEEP_DISABLED)

### Impacto de activar cada flag

Para construir o risk assessment (AC6), considerar:
- `execution.enabled: true` → SDC e Brownfield executam automaticamente quando triggered
- `qa.enabled: true` → QA Loop inicia automaticamente após @dev completa
- `specPipeline.enabled: true` → Spec Pipeline activa para todas as features novas

Risco principal: automação pode interferir com sessões interactivas onde o utilizador quer controlo manual.

### Rollback plan (per EPIC risk mitigation)

```bash
# Reverter mudança de config:
git revert <commit-hash-da-mudanca>
# Ou editar directamente core-config.yaml e redefinir flags para false
```

### Testing

- **Validação e2e:** Para cada workflow activado, executar uma run completa (ex: SDC com uma story de teste)
- **Regression:** `npm test` deve passar antes e depois da mudança de config
- **Não usar** `node --test` directamente nos workflow scripts (L1/L2 — read-only)
- **Evidence:** Guardar log de execução em `docs/qa/workflow-activation-evidence.md`
