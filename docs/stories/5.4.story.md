# Story 5.4: Agent Activation Automation — Enforce Skill File Loading

## Summary

**Problem:** Agent activation currently relies on manual discipline. When `/AIOX:agents:pm` is invoked, the legacy shim returns a file path but provides NO guarantee that the agent will:
1. Read the skill file completely
2. Execute the activation-instructions
3. Load handoff context from `.aiox/handoffs/`
4. Populate session state from previous work

**Result:** Agents can bypass the entire activation protocol by answering questions directly instead of following the formal process. This breaks context continuity and defeats the purpose of the skill-based activation system.

**Solution:** Build automation hooks + validation gates that ENFORCE skill file loading on agent activation.

**Impact:** CRITICAL — Restores agent protocol reliability and ensures every agent starts with full context state.

---

## Problem Statement

### Current Behavior (Broken)
```
User invokes: /AIOX:agents:pm "quero verificar hooks..."
↓
Legacy shim returns: "Read .claude/skills/AIOX/agents/pm/SKILL.md and execute activation-instructions"
↓
Agent choice 1 (correct): Read file → execute protocol → load context ✅
Agent choice 2 (broken): Ignore file → answer question directly ❌
↓
Result: No validation, no enforcement, success depends on agent discipline
```

### Root Cause
- Legacy shim is informational only (no execution guarantees)
- No hook fires on agent activation
- No gate validates that skill file was loaded
- Handoff loading is optional, not required

## Acceptance Criteria

- [ ] **AC1:** `UserPromptSubmit` hook extension (`.claude/hooks/agent-activation-enforce.cjs`, registered alongside `agent-activation-tracker.cjs`) intercepts prompts matching `@{agent}` or `/AIOX:agents:{agent}` and:
  - Detects agent name from the prompt text using the same `AGENT_ACTIVATION_PATTERN` regex as `agent-activation-tracker.cjs`
  - Identifies matching skill file (`.claude/skills/AIOX/agents/{name}/SKILL.md`)
  - Validates file exists AND is readable
  - If skill file is missing or unreadable → **WARN-and-proceed** (log to `.aiox/context-engine.log`, write `session.active_agent.skill_file_warning` to `hook-metrics.json`); activation is NOT blocked, consistent with the graceful-degradation pattern used by every other gate (Art. III/V `warn-and-proceed`)
  - **Note:** `/AIOX:agents:*` is a user-typed prompt, not a tool call — `PreToolUse`/`PostToolUse` hooks NEVER fire for agent activation. `UserPromptSubmit` (prompt-time) and `Stop` (response-end checkpoint) are the only correct interception points.

- [ ] **AC2:** Skill file loader (new utility `.claude/utils/skill-file-loader.js`) that:
  - Reads skill file completely
  - Parses YAML frontmatter (name, type, activation_type)
  - Extracts activation-instructions section
  - Returns structured object: `{ name, instructions, handoffPath, greeting }`

- [ ] **AC3:** New handoff auto-loader utility (`.claude/utils/handoff-loader.js` — file does NOT currently exist; `.claude/utils/` is empty, so this is a NEW file, not an enhancement):
  - On activation (called from the `UserPromptSubmit` hook in AC1), scans `.aiox/handoffs/*.json`
  - Handoff filenames follow the real format `handoff-{epochMs}-{storyId}.json` and are **keyed by `story_id`**, not by agent ID (confirmed structure: `{ timestamp, story_id, story_path, story_status, story_effort, agent, action, notes }`)
  - Resolves the active story from `core-config.yaml` / current sprint context (or accepts an explicit `storyId` argument); loads the **most recent handoff JSON whose `story_id` matches** (highest `epochMs` in filename, or latest `timestamp` field)
  - Injects the loaded JSON into session context as `AGENT_HANDOFF`
  - **No `consumed` field exists in this data model and none is added** — handoffs are immutable history records (consistent with `handoff-consolidation.md`); "already seen" tracking, if needed, is done separately via `session.active_agent.handoff_loaded_at` timestamp in `hook-metrics.json`, not by mutating the handoff file
  - If no matching handoff exists → WARN-and-proceed (no `AGENT_HANDOFF` injected, no error)

- [ ] **AC4:** Activation checkpoint (new `.claude/hooks/agent-activation-checkpoint.cjs`, registered on the `Stop` event as a response-end checkpoint — NOT `PostToolUse`, which does not fire for prompt-driven agent activation):
  - After the agent's response completes, verify that:
    - Skill file was loaded (check `session.active_agent.skill_file_loaded` in `hook-metrics.json`)
    - Handoff was considered (check `session.active_agent.handoff_loaded_at` or absence-with-warning in `hook-metrics.json`)
    - Greeting was displayed (check `.aiox/context-engine.log` for activation entry)
  - If any checkpoint missed → **WARNING** logged + remediation suggestion recorded for next turn (auto-reload context on the next `UserPromptSubmit`); never blocks the current response

- [ ] **AC5:** Session state injection (hook writes to `.synapse/metrics/hook-metrics.json`):
  - Field: `session.active_agent.skill_file_loaded` = timestamp
  - Field: `session.active_agent.handoff_consumed` = handoff ID (if any)
  - Field: `session.active_agent.activation_protocol_version` = "v1"
  - Persists across agent switches

- [ ] **AC6:** Tests validate:
  - Hook detects missing skill file and blocks
  - Skill file loader extracts instructions correctly
  - Handoff auto-loader finds and injects context
  - Checkpoint detects protocol violations
  - No regressions to existing agent activation

- [ ] **AC7:** Documentation updated:
  - `.claude/rules/agent-authority.md` — add "Skill File Loading" rule
  - `.claude/rules/agent-handoff.md` — clarify auto-loading behavior
  - `.claude/hooks/README.md` — document new hooks

## Scope

### IN (What We're Building)
1. Pre-execution hook to validate skill file existence
2. Skill file loader utility (parse + extract activation-instructions)
3. Auto-loading of handoff context from `.aiox/handoffs/`
4. Activation checkpoint to verify protocol was followed
5. Session state tracking (which agent, when, with what context)
6. Tests for all new utilities

### OUT (What We're NOT Doing)
- Rewriting legacy shim (keep as-is)
- Changing skill file format (YAML frontmatter + instructions)
- Moving skill files (stay in `.claude/skills/AIOX/agents/*/`)
- Blocking users from direct agent commands (enforcement only, not restrictive)

### Hook Execution Order

```
User submits prompt: "@pm <args>" or "/AIOX:agents:pm <args>"
  ↓
[USER-PROMPT-SUBMIT] agent-activation-tracker.cjs (existing, unchanged)
  → Detects active agent via AGENT_ACTIVATION_PATTERN
  → Writes session.active_agent.id to hook-metrics.json
  → Runs Phases 1-4 context engine (existing, Story 5.3.x)
  ↓
[USER-PROMPT-SUBMIT] agent-activation-enforce.cjs (NEW — chained after tracker)
  → Re-uses detected agent name
  → Validates .claude/skills/AIOX/agents/{name}/SKILL.md exists and is readable
    → Missing/unreadable: WARN-and-proceed (log + hook-metrics.json field), do NOT block
  → Loads skill file via skill-file-loader.js → { name, instructions, handoffPath, greeting }
  → Calls handoff-loader.js → finds latest .aiox/handoffs/handoff-{epochMs}-{storyId}.json
    for the active story; if none found, WARN-and-proceed (no AGENT_HANDOFF)
  → Writes session.active_agent.skill_file_loaded (timestamp) and
    session.active_agent.handoff_loaded_at (timestamp or null) to hook-metrics.json
  → Returns additionalContext (skill instructions + AGENT_HANDOFF) for the prompt
  ↓
Agent (Claude) receives skill file content + handoff context inline in prompt context
Agent executes activation-instructions, displays greeting
  ↓
[STOP] agent-activation-checkpoint.cjs (NEW — fires when the response completes)
  → Reads hook-metrics.json: checks skill_file_loaded, handoff_loaded_at present
  → Checks .aiox/context-engine.log for activation/greeting entry
  → If any checkpoint missing → WARNING logged + flag set for remediation on next
    UserPromptSubmit (auto-reload); never blocks or alters the completed response
```

## Rationale

**Why this matters:**
- Current system has **no enforcement** of agent protocol
- Session context breaks when agents skip skill file loading
- Handoff protocol only works if agent manually reads handoff file
- Leads to confusion, lost context, repeated work

**Why this approach:**
- `agent-activation-tracker.cjs` already runs on `UserPromptSubmit` and is the proven interception point for agent detection — this story chains additional logic onto the same event rather than inventing a new mechanism
- No changes to skill file format or legacy shim
- WARN-and-proceed (never block) — consistent with the graceful-degradation pattern used by every other gate in `enforcement-gates.md` (Art. III/V `warn-and-proceed`) and by `agent-activation-tracker.cjs` itself
- Auto-loading reduces manual steps without forcing behavioral change
