# Story 8.1.5: Real-time Monitoring

**Status:** Ready
**Story ID:** 8.1.5
**Created:** 2026-06-11

**PRD Source:** `docs/prd/epic-8/phase-1-observability.md#815-real-time-monitoring-2sp`
**Story Points:** 2

---

## Story

**As an** external monitoring tool operator (e.g., Datadog, New Relic integrator),
**I want** a webhook endpoint that pushes gate decisions and story updates in real time,
**so that** external systems can subscribe to AIOX framework events without polling.

---

## Acceptance Criteria

1. Webhook endpoint: `POST /api/webhooks/gate-event`
2. Payload: Gate decision (article, verdict, timestamp, agent)
3. Subscription API: Store webhook URLs in PostgreSQL
4. Testing: Trigger gate → webhook fires within 5s
5. Graceful degradation: Missing webhook doesn't block gate

**Use Case:** External monitoring tools (Datadog, New Relic) can subscribe

---

  - [ ] PostgreSQL table for webhook subscriptions: `webhook_subscriptions` (id, url, created_at, active, last_delivery_status)
  - [ ] CRUD endpoints: register/list/delete webhook URLs (auth via `METRICS_API_KEY`, consistent with 8.1.4)
- [ ] Task 3 (AC: 4) — Wire dispatch into gate decision flow
  - [ ] After a gate decision is logged (per `.claude/hooks/lib/gate-logger.cjs`), asynchronously dispatch the event payload to all active subscriptions
  - [ ] Verify end-to-end: trigger a gate decision → subscriber receives payload within 5s
- [ ] Task 4 (AC: 5) — Graceful degradation
  - [ ] Wrap webhook dispatch in try/catch with timeout; failures logged (e.g., to `last_delivery_status`) but never thrown back into the gate decision path
  - [ ] Confirm via test: simulate subscriber down/unreachable → gate decision still completes normally and logs correctly

## Dev Notes

### Source Documents

- PRD: `docs/prd/epic-8/phase-1-observability.md#815-real-time-monitoring-2sp`
- Gate decision flow: `.claude/rules/enforcement-gates.md` — describes the hook layer (`PreToolUse`), `gate-logger.cjs`, and the graceful degradation principle this story must respect
- Dashboard backend (8.1.4): provides PostgreSQL connection + auth pattern to reuse

### Critical Constraint — Do Not Modify Gate Hooks Directly

The constitutional enforcement gates (`.claude/hooks/enforce-*.cjs`) are part of the framework's enforcement layer. This story should integrate by:
- Observing gate-log writes (e.g., via `.aiox/gate-logs/*.jsonl` tailing, or a hook into `gate-logger.cjs`'s logging function) rather than adding new logic inline in the enforcement decision path
- Any change to `.claude/hooks/` files must preserve the existing decision priority (`Art. II > Art. III > Art. IV > Art. V > Art. VI-VII`) and never add latency to the blocking decision itself

### File Locations

- New: `/api/webhooks/gate-event` route (or dispatcher module if this is implemented as outbound-only)
- New: PostgreSQL table `webhook_subscriptions` (add to migration — coordinate with 8.1.7's `migrations/001-metrics-init.sql` or create a follow-up migration `002-webhook-subscriptions.sql`)
- Reference (read-only): `.claude/hooks/lib/gate-logger.cjs`, `.claude/rules/enforcement-gates.md`

### Testing

- Unit: webhook payload construction from a gate-log entry
- Unit: subscription CRUD (register, list, delete) with auth checks
- Integration: trigger a real gate decision (e.g., a test `git push` scenario per enforcement test patterns in `tests/hooks/enforcement.test.js`) and verify webhook fires within 5s (AC4)
- Integration: simulate unreachable subscriber URL, verify gate decision is unaffected (AC5) — this is the most important test in this story
- Reference existing test pattern: `tests/hooks/enforcement.test.js` (covers every gate using `node:test`)
