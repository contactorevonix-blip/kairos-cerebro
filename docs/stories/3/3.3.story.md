# Story 3.3 — Corruption Recovery & Backup Export

**Epic:** EPIC-3 (Data Durability)  
**Status:** Draft  
**Complexity:** 10sp  
**Assignee:** @data-engineer  
**Created:** 2026-06-04

---

## Summary

Add recovery mechanism for SQLite database corruption. On startup, run `PRAGMA integrity_check`. If fails, restore from last known-good backup. Implement daily backup export job to `.aiox/backups/monitor-{YYYY-MM-DD}.db.gz` with checksums.

**Why:** NFR3 requires durability. Need to recover from hardware failures or corruption without losing data.

---

## Acceptance Criteria

- [ ] Create `src/db/Recovery.ts`:
  - On app startup, run `PRAGMA integrity_check`
  - If fails (returns corruption list), log to audit + stop app
  - User must manually restore from `.aiox/backups/` via `kairos-monitor restore --date 2026-06-03`

- [ ] Create `src/db/Backup.ts`:
  - Daily backup job at 04:00 UTC
  - Copies `.aiox/monitor.db` to `.aiox/backups/monitor-{YYYY-MM-DD}.db.gz`
  - Compresses with gzip
  - Generates SHA256 checksum → `.aiox/backups/monitor-{YYYY-MM-DD}.sha256`
  - Keeps last 10 backups (older ones deleted)

- [ ] Backup schema in `.aiox/BACKUP-MANIFEST.json`:
  ```json
  {
    "backups": [
      {
        "date": "2026-06-03",
        "filename": "monitor-2026-06-03.db.gz",
        "size_bytes": 1024000,
        "checksum_sha256": "abc123...",
        "created_at": "ISO8601",
        "verified": true
      }
    ]
  }
  ```

- [ ] Restore CLI command:
  ```bash
  kairos-monitor restore --date 2026-06-03
  # Restores from backup, verifies checksum, restarts Monitor
  ```

- [ ] Verification: after backup creation, verify checksum matches
- [ ] Tests: simulate corruption, verify recovery, checksum validation

---

## Implementation Notes

- **No hot backup:** backup is offline copy (app must be running, DB not locked)
- **Compression:** gzip reduces backup size 5-10x
- **Manifest:** tracks all backups for audit + recovery decision-making
