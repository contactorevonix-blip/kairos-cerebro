# Story 3.2 — Rollup Strategy (Hourly → Daily → Monthly)

**Epic:** EPIC-3 (Data Durability)  
**Status:** Draft  
**Complexity:** 10sp  
**Assignee:** @data-engineer  
**Created:** 2026-06-04

---

## Summary

Implement rollup jobs to compress older metrics data. Hourly metrics → daily aggregates (avg, p95, max). Daily → monthly archives. Keeps recent data hot (fast queries), old data compressed (cheap storage). Runs daily at 02:00 UTC.

**Why:** NFR3 requires <1GB disk per 30 days. Raw metrics at per-session granularity would exceed this. Rollup compresses old data.

---

## Acceptance Criteria

- [ ] Create `src/db/Rollup.ts`:
  - Hourly job: aggregate `metrics` table entries older than 1h
  - Insert aggregated rows into `metrics_daily` table
  - Delete aggregated raw rows (same entry can't exist in both)

- [ ] Aggregation formula:
  ```sql
  SELECT
    session_id,
    DATE(timestamp) as rollup_date,
    AVG(token_input) as avg_input,
    PERCENTILE_CONT(0.95) WITHIN GROUP (ORDER BY token_input) as p95_input,
    MAX(token_input) as max_input,
    SUM(token_input) as total_input
    -- [similar for token_output, prompt_count]
  FROM metrics
  WHERE timestamp < DATETIME('now', '-1 hour')
  GROUP BY session_id, rollup_date;
  ```

- [ ] Monthly rollup: daily → monthly in `metrics_monthly` table
  - Runs on day 1 of each month at 03:00 UTC
  - Aggregates previous month only

- [ ] Retention schedule:
  - Raw metrics: 7 days
  - Daily aggregates: 90 days
  - Monthly: 1 year (archive after)

- [ ] Delete job: aged-out data is deleted on schedule
- [ ] Tests: verify aggregation accuracy, retention schedule, deletion

---

## Implementation Notes

- **Idempotency:** rollup is safe to run multiple times (uses timestamp check)
- **Parallelism:** can run while Monitor API is serving (read-only aggregation)
- **Disk savings:** typical 10:1 compression ratio (raw → aggregate)
