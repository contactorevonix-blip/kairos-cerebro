# Story 12.G3 — Spec-Critique Verdict Logic Tests

## Description

`.aiox-core/development/tasks/spec-critique.md` defines a deterministic
verdict-calculation contract across 5 weighted dimensions (Accuracy 25%,
Completeness 25%, Consistency 20%, Feasibility 15%, Alignment 15%) and an exact
verdict rule set:

- **APPROVED**: no HIGH severity issues, average score >= 4.0, all dimensions >= 3
- **NEEDS_REVISION**: has MEDIUM severity issues OR average 3.0-3.9 OR any dimension < 3 (with no HIGH issues)
- **BLOCKED**: has HIGH severity issues OR average < 3.0 OR any dimension <= 1

This logic is currently exercised only by an LLM (the @qa agent) running the
critique task manually — there is no automated test harness asserting the
verdict rules behave correctly at the boundaries. EPIC-9 delivered 84
automated tests with full coverage for its scope; EPIC-12's spec-critique
process has zero automated coverage of its own verdict logic, which is a risk:
if the verdict thresholds are ever encoded into a script/hook (e.g. to support
12.G2's automated gate), an incorrect boundary condition could silently let a
BLOCKED-quality spec through as APPROVED.

This story creates a standalone test suite (input fixtures + assertions) that
exercises the verdict logic exactly as specified in `spec-critique.md`,
independent of whether the calculation is performed by an LLM or by code. The
suite acts as an executable specification: any future automation of this logic
(by @dev or @architect) can run against these fixtures to confirm conformance.

## Dev Notes

### Source of truth — do not invent thresholds

All verdict thresholds, dimension weights, and the exact verdict rule text
MUST be taken verbatim from `.aiox-core/development/tasks/spec-critique.md`
(section `## Verdict Logic`, lines defining `verdict_rules.APPROVED`,
`verdict_rules.NEEDS_REVISION`, `verdict_rules.BLOCKED`). Do not approximate or
re-derive these thresholds — copy them exactly:

```yaml
APPROVED:
  condition: |
    - No HIGH severity issues
    - Average score >= 4.0
    - All dimensions >= 3

NEEDS_REVISION:
  condition: |
    - Has MEDIUM severity issues OR
    - Average score between 3.0-3.9 OR
    - Any dimension < 3 but no HIGH issues

BLOCKED:
  condition: |
    - Has HIGH severity issues OR
    - Average score < 3.0 OR
    - Any dimension <= 1
```

### critique.json schema reference

Use the JSON Schema embedded in `spec-critique.md`'s `## Output Schema`
section as the fixture shape contract: `storyId`, `critiquedAt`, `verdict`,
`verdictReason`, `scores: { accuracy, completeness, consistency, feasibility,
alignment, average }`, `issues: [{ id, severity, category, description, ... }]`.

A real example exists at `docs/prd/process-mapper/critique.json` — read it to
confirm field-naming conventions before writing fixtures (e.g., confirm
`average` vs `avgScore`, confirm `severity` enum casing).

### Important clarification on AC3 Case D/E logic

The rule set combines conditions with AND (within a verdict) and the verdicts
are evaluated in priority order (the task likely evaluates BLOCKED conditions
first, then APPROVED, defaulting to NEEDS_REVISION — verify the exact
evaluation order against `spec-critique.md`'s `## Execution Flow` → `Step 4:
Calculate Verdict` before finalizing Case E's expected outcome, since "any
dimension < 3" is explicitly called out under NEEDS_REVISION's OR-list, not
BLOCKED's (BLOCKED only triggers on "any dimension <= 1"). Case E (one
dimension = 2.9) is NEEDS_REVISION, not BLOCKED, because 2.9 > 1. Confirm this
distinction is captured precisely in the test assertion comment.
