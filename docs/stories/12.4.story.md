# Story 12.4 — No-Invention Gate & Requirement Traceability (Art. IV)

## Description

Enforce Specification Requirement Traceability (FR-5.4, Art. IV). Extend the
no-invention gate with a second validation layer that checks cited
FR-*/NFR-*/CON-* references actually exist in requirements.json, so that
pattern detection plus reference validation together produce zero false
positives across all spec.md statements.

### Completion Notes
- The Art. IV gate is now genuinely two-layer:
  - **Layer 1** (`enforce-no-invention.cjs`): blocks normative statements (MUST/SHALL/REQUIRED) carrying **no** reference.
  - **Layer 2** (`enforce-spec-reference-validation.cjs`): blocks statements that cite a reference which does **not exist** in `requirements.json` (a "dangling" / fabricated citation). This closes the false-negative where L1 alone is fooled by an invented `[FR-99.9]` citation.
- Hook discovers `requirements.json` by walking up from the spec file's directory (checking each dir and a sibling `spec/`), bounded to the project root and depth 12. Confirmed against the real `docs/stories/epics/EPIC-12/spec/requirements.json`.
- **Graceful degradation (CON-3):** when no readable `requirements.json` is found, the gate emits `warn-and-proceed` and never blocks. `AIOX_SPEC_REF_PERMISSIVE=1` downgrades dangling-reference blocks to a warning for pre-requirements drafts.
- References inside fenced code blocks (` ``` `) are ignored so example snippets aren't validated.
- AC4 clarification: there is no authored EPIC-12 `spec.md` body with "150 statements" in this story's scope, and no `complexity.json`/`research.json` were present for cross-checking. AC4 is delivered as a **mechanism guarantee** — any spec.md written from now on is forced into 100% requirements.json traceability by L1+L2. Reference resolution is restricted to `requirements.json` (the authoritative id set); complexity.json/research.json citations use the `[research:]`/`[finding:]` form already handled by Layer 1.
