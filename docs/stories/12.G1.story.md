# Story 12.G1 — Hook Cross-Reference Validation (Art. IV)

## Description

`enforce-no-invention.cjs` (the Article IV — No Invention gate) detects RFC-2119
normative keywords (MUST / MUST NOT / SHALL / SHALL NOT / REQUIRED) and checks
that the line carries a reference token matching the pattern
`FR-\d+ | NFR-\d+ | CON-\d+ | [research:...] | [finding:...]`. It does **not**
verify that the cited token actually exists in the story's `requirements.json`.

Example: a spec line `"The system MUST log all FR-99 events."` passes the
current gate because it has both a MUST keyword and an `FR-99` token — even if
`FR-99` was never defined in `requirements.json`. This is a hole in Article IV
(No Invention) enforcement: an agent (or human) can fabricate a requirement ID
and the existing hook will not catch it.

This story adds a second, complementary gate that runs after
`enforce-no-invention.cjs` passes, and validates that every `FR-*` / `NFR-*` /
`CON-*` token cited in `spec.md` resolves to an entry that actually exists in
the sibling `requirements.json`.

## Acceptance Criteria

1. [ ] AC1: New hook file `.claude/hooks/enforce-spec-reference-validation.cjs` created.
2. [ ] AC2: Matcher = `Write`/`Edit` on `*spec.md`, registered to run AFTER `enforce-no-invention.cjs` passes (i.e., does not duplicate or replace the existing Art. IV keyword/reference-pattern gate — it is an additional, narrower check).
3. [ ] AC3: Hook reads the sibling `requirements.json` (same directory as the target `spec.md`, e.g. `docs/stories/{storyId}/spec/requirements.json`) if it exists, and extracts the full list of valid `FR-*`, `NFR-*`, `CON-*` IDs from it.
4. [ ] AC4: Hook scans `spec.md` content (the `content`/`new_string` being written) and extracts every cited reference matching the regex `(FR|NFR|CON)-\d+`.
5. [ ] AC5: Hook BLOCKs the operation if any cited reference is not found in the `requirements.json` ID list. Block message must be specific and actionable, e.g.: `"FR-99 cited but not found in requirements.json"` (one line per missing reference).
6. [ ] AC6: Hook registered in `.claude/settings.json` under the appropriate `PreToolUse` matcher block for `Write` and `Edit` (mirroring how `enforce-no-invention.cjs` is registered).
7. [ ] AC7: Tests added to `tests/hooks/enforcement.test.js` covering 3 cases: (a) valid reference that exists in requirements.json → allow, (b) reference that does not exist in requirements.json → block, (c) malformed/missing requirements.json → graceful degradation (warn-and-proceed, never a hard crash).
8. [ ] AC8: Metrics recorded in `.synapse/metrics/hook-metrics.json` under a new key `art-iv-reference-validation: { checked, validated, blocked }`, following the existing `gl.recordMetrics()` pattern used by `enforce-no-invention.cjs`.

### Relevant existing code (read before implementing)

- `.claude/hooks/enforce-no-invention.cjs` — the sibling Art. IV gate. Reuse its
  patterns directly:
  - `gl.parseInput(gl.readStdin())` to read the PreToolUse payload
  - `extractContent(toolInput)` helper (or an equivalent) to pull `content` /
    `new_string` from `tool_input`
  - `gl.recordMetrics({...})`, `gl.logGateDecision({...})`, `gl.emitDecision('deny', reason)`
    for the allow/block/log contract
  - `process.exitCode = 2` on block (matches existing convention)
  - `AIOX_NO_INVENTION_PERMISSIVE`-style env var pattern is a useful precedent
    for graceful degradation if requirements.json is missing — this new gate
    should warn-and-proceed (not block) when `requirements.json` does not exist
    at all, since not every spec necessarily has one yet (e.g., pre-Gather
    phase drafts). Only BLOCK when requirements.json exists AND a cited
    reference is missing from it.
- `.claude/hooks/lib/gate-logger.cjs` — shared utility (`recordMetrics`,
  `logGateDecision`, `parseInput`, `readStdin`, `emitDecision`). Update this
  file only to register the new metric key if the schema requires explicit
  registration (verify before editing — most gates just call
  `recordMetrics({ key: value })` and the file merges idempotently per
  `.claude/rules/enforcement-gates.md`).
- `.claude/rules/enforcement-gates.md` — describes the existing gate inventory
  table and decision-log format (`.aiox/gate-logs/{article}-{YYYY-MM-DD}.jsonl`).
  This new gate should follow the same JSONL decision-log convention.
- `docs/stories/epics/EPIC-12/spec/requirements.json` — a real example of the
  requirements.json shape produced by Phase 1 (Gather) of the Spec Pipeline.
  Read this file to confirm the exact ID field name/format (e.g., `id: "FR-1"`)
  before writing the extraction logic — do not assume the schema, verify it.

### Why this gate is separate from enforce-no-invention.cjs

`enforce-no-invention.cjs` checks the **pattern** of a reference (does the line
look traceable). This new hook checks the **existence** of the reference
(does the cited ID actually exist in requirements.json). Keeping them as two
hooks preserves single-responsibility and matches the existing AIOX pattern of
one `.cjs` file per concern under `.claude/hooks/`.
