## Story

**As a** user typing `*brief` or `*dev`,
**I want** the matching command block injected into context,
**so that** response-mode switching actually takes effect.

## Epic Context

## Acceptance Criteria

1. Given a prompt containing `*brief`, when the engine runs, then L7 injects the `*brief` command block's rules (e.g., "Use bullet points only", or whatever the current block content is).
2. Given a prompt with no `*command` token, when the engine runs, then L7 is skipped (0 rules) without affecting other layers.
3. Given the current `.synapse/commands` file, when the L7 parser runs, then it extracts at least one rule for `*brief` — the `#`-comment-header stripping bug (`l7-star-command.js:139-166`) is resolved.
4. Multiple commands in one prompt (e.g., `*brief *dev`) each inject their respective block (both present in merged output).
5. L7 rules are treated as style-only in conflict resolution: they cannot override L0 constitutional rules or L2 agent authority rules (verified against 82.2 precedence table — L7 precedence = 10, lowest after L1).

### Root Cause of the L7 Bug (PRD §0, verified)

```
.synapse/commands  current format:
  # [*brief]
  Use bullet points only.
  Keep responses concise.
  # [*dev]
  ...

loadDomainFile() strips lines starting with '#' (comments).
_parseCommandBlocks() in l7-star-command.js:139-166 looks for [*X] delimiters.
After stripping, the delimiters are gone → parser finds 0 command blocks.
```

**Fix option A (L3):** Change `# [*brief]` to `[*brief]` (remove the `#` prefix). If `loadDomainFile` only strips lines that are ONLY `#` (i.e., blank comment lines), this may work. Inspect `loadDomainFile` behaviour before committing.

**Fix option B (L1):** Make the parser understand `# [*brief]` as a delimiter. Risk: changes L1 parser behaviour for all downstream consumers. Prefer option A.
