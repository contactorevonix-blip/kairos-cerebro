# Story 4.2 — Bearer Token Lifecycle Management

## Summary

Implement Bearer token lifecycle: generate on first run, store in `.clauderc`, rotate on demand, revoke old tokens. Tokens are simple format (e.g., `sk-{random-32-bytes-hex}`). Only required for API access (NFR4: Bearer token auth for localhost).

**Why:** NFR4 requires auth. Bearer tokens are minimal, stateless, localhost-only.

---

## Acceptance Criteria

- [ ] Create `src/auth/TokenManager.ts`:
  - Generate token on app startup if none exists
  - Token format: `sk-{crypto.randomBytes(32).toString('hex')}`
  - Store in `.clauderc` under `[kairos-monitor] token_bearer = ...`

- [ ] CLI commands:
  ```bash
  kairos-monitor token list
    → show current token (last 8 chars masked)
    
  kairos-monitor token rotate
    → generate new token, save to .clauderc, show new token once
    
  kairos-monitor token revoke <token>
    → revoke a token (for multi-user setups, future)
  ```

- [ ] Validation:
  - Token must be present on every API request
  - Malformed tokens rejected (must start with `sk-` or match pattern)
  - Expired tokens: not implemented (tokens never expire, but revoke for security events)

- [ ] Security:
  - Never log full token (only last 8 chars)
  - Token stored in `.clauderc` file (not in env, not in memory longer than needed)
  - On rotate, old token invalid immediately

- [ ] Tests: verify token generation, storage, validation, rotation

---

## Implementation Notes

- **Stateless:** tokens are just random strings, no database lookup needed
- **Localhost only:** bearer token is enough, no full OAuth needed
- **One token per instance:** Monitor server has one active token at a time
