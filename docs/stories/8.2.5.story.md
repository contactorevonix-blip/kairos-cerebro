# Story 8.2.5: Creation Justification Validator

**Status:** Ready
**Story ID:** 8.2.5
**Created:** 2026-06-11
**Epic:** EPIC-8 — Framework Evolution (Phase 2: IDS Enhancement)
**PRD Source:** `docs/prd/epic-8/phase-2-ids.md#825-creation-justification-validator-15sp`

## Story

**As a** Product Owner running `*validate-story-draft`,
**I want** the validator to require and check a REUSE>ADAPT>CREATE justification (`created_reason`, `evaluated_patterns`, `rejection_reasons`, `new_capability`) for any story that introduces a new entity,
**so that** Article IV-A ("No new entity without REUSE>ADAPT>CREATE justification") is enforced automatically instead of relying on agents remembering to document it.

---

## Acceptance Criteria

1. Gate integration: `*validate-story-draft` checks for created entities
2. Validation rule: New entity in story must include:
   - `created_reason: "..."` (why not REUSE/ADAPT existing?)
   - `evaluated_patterns: [list of existing entities considered]`
   - `rejection_reasons: {pattern: reason}`
   - `new_capability: "..."` (unique capability this provides)
3. Registry update: Auto-register new entities with justification metadata
4. Audit trail: Justifications stored in entity-registry.yaml under `metadata.created_justification`

## Risks

- **Risk:** Retrofitting this validator onto `*validate-story-draft` (an existing @po task) could break the existing 10-point validation checklist (`.claude/rules/story-lifecycle.md`) if not additive.
- **Mitigation:** AC1 says "checks for created entities" — this should be an additional conditional check (only triggers when a story introduces a new entity), not a replacement of the existing 10-point checklist. Implement as an additive 11th check or a sub-check within an existing category (e.g., category 2 "Technical Implementation Guidance").

- **Risk:** AC4 stores justification under `entity-registry.yaml`'s `metadata.created_justification` — the current registry schema (per the 823-entity sample) has `metadata` only at the top (file-level) `metadata:` block (version, lastUpdated, entityCount, etc.), not per-entity. Per-entity blocks currently have fields like `path`, `type`, `purpose`, `keywords`, `usedBy`, `dependencies`, `adaptability`, `checksum`, `lastVerified` — no `metadata` sub-key per entity was observed.
- **Mitigation:** AC4 likely intends a new per-entity `metadata.created_justification` block (i.e., add a `metadata:` key to individual entity entries, distinct from the file-level `metadata:`). @dev should confirm this interpretation with @architect before implementing, since it's a schema addition affecting all future entity registrations (and potentially 8.2.2's auto-heal "auto-add" logic, which should also populate this field for newly-detected entities going forward).
