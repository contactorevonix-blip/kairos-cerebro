# Story 5.1 — Hook Automation Audit

**Epic:** AIOX-OPS (AIOX Operacional 100%) | **Story ID:** 5.1 | **Effort:** 8sp | **Status:** Ready | **Executor:** @dev | **Quality Gate:** @qa

## Goal

Auditar os 25 hooks registados em `.claude/settings.json` e verificar — por smoke test real, não por afirmação — que cada um existe, executa sem erro com input simulado, e completa dentro do timeout configurado. Produzir uma matriz pass/fail que se torna a base de verdade para as restantes stories do epic.

## Contexto / Problema

GAP G1 (ver EPIC-AIOX-OPERACIONAL.md, Secção 3): existem 25 hooks registados em `settings.json` (PreToolUse, PostToolUse, UserPromptSubmit, Stop, PreCompact, PostCompact, SessionStart, SubagentStop, TaskCompleted, ConfigChange). Sessão após sessão o STATE.md **afirma** que os hooks funcionam, mas nunca houve auditoria em runtime. "Operacional" exige verificação, não assunção (Art. IV — No Invention).

## Acceptance Criteria

- [ ] **AC1:** Inventário completo dos hooks registados em `settings.json` produzido — cada hook mapeado a (evento, matcher, ficheiro, timeout, async sim/não). Confirmar que o ficheiro de cada hook existe no disco.
  - *Given* `settings.json` com 25 hooks, *When* o inventário corre, *Then* cada entrada lista evento/matcher/ficheiro/timeout e marca `file_exists: true/false`.
- [ ] **AC2:** Cada hook executado com input simulado (stdin payload representativo do seu evento). Capturar exit code, stderr e duração.
  - *Given* um hook, *When* executado com payload de evento simulado, *Then* regista `exit_code`, `stderr` (vazio = pass), `duration_ms`.
- [ ] **AC3:** Validação de timeout — para cada hook, `duration_ms` comparado com o `timeout` configurado em `settings.json`. Flag `WARN` se `duration_ms > 0.8 * timeout`.
  - *Given* duração medida e timeout configurado, *When* comparados, *Then* marca PASS / WARN (perto do limite) / FAIL (excede).
- [ ] **AC4:** Matriz pass/fail gerada com veredicto por hook (PASS / WARN / FAIL) e razão. Foco explícito nos hooks de enforcement (Art. II-VII), `synapse-engine.cjs`, `agent-activation-tracker.cjs`.
- [ ] **AC5:** Relatório `docs/qa/HOOK-AUDIT-REPORT.md` (humano) + `docs/qa/hook-audit-results.json` (machine-readable) gerados. Hooks FAIL listados como GAPs accionáveis para 5.2/5.4/5.5/5.6.

## Scope

**IN:**
- Os 25 hooks registados em `.claude/settings.json` (todos os eventos)
- Smoke test com input simulado por evento
- Matriz pass/fail + relatórios (md + json)

**OUT:**
- Corrigir hooks partidos (isso é trabalho de 5.2/5.4/5.5/5.6 conforme o output)
- Modificar `settings.json` (auditoria é read-only sobre a config)
- Hooks em `~/.claude/helpers/` que são globais do utilizador (fora do repo) — listar mas marcar `scope: user-global, out-of-audit`
- Modificar L1/L2 (`.aiox-core/core/`, `.aiox-core/development/`)

## Dependencies

- **Prerequisito:** Nenhum — esta story arranca primeiro no epic (caminho crítico).
- **Bloqueia:** 5.4, 5.5, 5.6 (consomem o output da auditoria), 5.7 (consolidação final).
- **Recursos:** `.claude/settings.json`, `.claude/hooks/*.cjs`, `.claude/hooks/*.js`.

## Complexity Estimate

8sp — 25 hooks × (inventário + execução simulada + medição). Complexidade média-alta: requer construir payloads simulados por tipo de evento e harness de execução. HIGH risk porque o output redefine o scope de 4 stories downstream.

## Business Value

Transforma "assumimos que os hooks funcionam" em "sabemos quais funcionam". É a fundação factual de todo o epic — sem esta auditoria, as stories 5.4/5.5/5.6 operam sobre suposições. Resolve a tensão recorrente do STATE.md ("configured but not operationalized") com evidência.

## Risks

- **R1 (MEDIUM):** Hooks async (`async: true`) podem não devolver exit code de forma fiável no harness → mitigação: testar invocação directa do script, não o pipeline async.
- **R2 (MEDIUM):** Payloads simulados podem não representar input real → mitigação: derivar payloads dos formatos documentados em cada hook + `agent-handoff.md`/`story-lifecycle.md`.
- **R3 (LOW):** Auditoria revela muitos FAIL, expandindo o scope do epic → mitigação: FAIL vira backlog accionável, não bloqueia esta story.
