# Story 5.4 — Agent Handoff Automation

**Epic:** AIOX-OPS (AIOX Operacional 100%) | **Story ID:** 5.4 | **Effort:** 5sp | **Status:** Draft | **Executor:** @dev | **Quality Gate:** @qa

## Story

**As a** AIOX pipeline,
**I want** agent transitions to automatically generate handoff artifacts and consolidate at the 5+ threshold without manual intervention,
**so that** context is preserved across agent switches without any human having to trigger consolidation manually.

## Goal

Validar que o hook `post-push-handoff-consolidate.js` funciona correctamente: (1) consolida automaticamente quando 5+ handoffs existem para o mesmo pipeline, (2) está registado em `.claude/settings.json` com matcher correcto, (3) preserva conteúdo crítico conforme `handoff-consolidation.md`, e (4) não dispara quando threshold < 5. Corrigir qualquer bug encontrado.

## Contexto / Problema

O hook `post-push-handoff-consolidate.js` **EXISTE** (entregue em Story 1.17, commit 196b578) e **ESTÁ REGISTADO** em `.claude/settings.json` L295. **O GAP real não é a ausência do hook, mas a falta de testes e validação de funcionamento.** Esta story valida que o hook consolidação funciona correctamente em cenários reais, corrige qualquer problema encontrado, e documenta qualquer comportamento inesperado identificado pela Story 5.1 (Hook Automation Audit).

**Racional:** A auditoria (AUDIT-REPORT.md) não descobriu que o hook existia porque não testou operações pós-push. Esta story fecha essa lacuna de validação operacional.

## Acceptance Criteria

- [ ] **AC1:** Hook `post-push-handoff-consolidate.js` validado funcionando — quando 5+ handoffs existem para o mesmo pipeline, o hook consolida automaticamente após git push.
  - *Given* 5+ handoff YAMLs para o mesmo pipeline + git push, *When* `post-push-handoff-consolidate.js` é invocado (PostToolUse), *Then* os handoffs mais antigos são arquivados em `.aiox/handoffs/_archive/` e RUN-LOG.md é criado/actualizado com conteúdo crítico preservado. Se falhar, corrigir o bug identificado.

- [ ] **AC2:** Hook verificado registado correctamente em `.claude/settings.json` (PostToolUse com matcher `Bash(git push*)`).
  - *Given* `.claude/settings.json`, *When* inspeccionado, *Then* `post-push-handoff-consolidate.js` está presente no evento PostToolUse com o matcher correcto. Se registo estiver errado, corrigir.

- [ ] **AC3:** Preservação de conteúdo crítico verificada — consolidação NÃO perde: blockers resolvidos, decisões arquitecturais, schema/migration changes, ficheiros criados/modificados por wave, open items para próxima sessão. [Fonte: `.claude/rules/handoff-consolidation.md` — "What MUST be Preserved"]
  - *Given* consolidação corre, *When* output é inspeccionado, *Then* todos os campos "MUST be Preserved" estão presentes no RUN-LOG.md resultante.

- [ ] **AC4:** Handoff mais recente permanece como YAML individual após consolidação — não é arquivado.
  - *Given* consolidação de 6 handoffs, *When* completa, *Then* `.aiox/handoffs/` contém exactamente 1 YAML (o mais recente) + o `_archive/` com os 5 anteriores.

- [ ] **AC5:** Testes de consolidação passam sem regressões. Se Story 5.2 (Script Lifecycle) criou testes em `tests/hooks/handoff-consolidation.test.js`, validar que esses testes passam. Se não existem testes de consolidação pós-push, criar `tests/hooks/handoff-automation.test.js` com cobertura: happy path (5+ handoffs → consolida), threshold (4 → não consolida), content preservation, most-recent-stays-as-YAML.
  - *Given* testes de consolidação, *When* `node --test tests/hooks/handoff-*.test.js` corre, *Then* 0 failures e rastreabilidade clara de qual story é responsável por qual test file.

- [ ] **AC6:** Output da Story 5.1 (Hook Automation Audit) consultado — se a auditoria revela outros hooks de handoff partidos, estes são documentados como GAPs accionáveis (não necessariamente corrigidos nesta story, mas registados).

## Scope

**IN:**
- Implementar `post-push-handoff-consolidate.js` (hook ausente identificado no AUDIT-REPORT.md G4)
- Registar o hook em `settings.json`
- Validar que transições de agente geram handoffs (smoke test manual ou automatizado)
- Testes automáticos para a lógica de consolidação
- Consultar output de Story 5.1 e documentar GAPs adicionais (se encontrados)

**OUT:**
- Reescrever a lógica de agent-handoff (apenas implementar o hook ausente)
- Modificar L1/L2 (`.aiox-core/core/`, `.aiox-core/development/`)
- Corrigir outros hooks partidos revelados por 5.1 (scope de 5.5/5.6)
- Implementar o RUN-LOG.md consolidation format (já definido em `handoff-consolidation.md`)

## Dependencies

- **Prerequisito:** Story 5.1 (Hook Automation Audit) — o output da auditoria confirma o estado actual dos hooks de handoff e pode revelar problemas adicionais que informam o scope desta story.
- **Bloqueia:** 5.7 (GAP Discovery & Fix — consolida outputs).
- **Recursos:** `.claude/rules/handoff-consolidation.md`, `.claude/rules/agent-handoff.md`, `.claude/settings.json`, `.claude/hooks/`, `tests/hooks/handoff-consolidation.test.js` (Story 1.20 — referência).

## Complexity Estimate

5sp — implementar 1 hook novo (consolidação) + registar + testes. Média complexidade: a lógica de consolidação já está documentada em `handoff-consolidation.md`; o trabalho principal é implementar fielmente a spec + testes.

## Business Value

Elimina o único caso documentado de "hook referenciado mas não existente" no sistema (AUDIT-REPORT.md G4). Com `post-push-handoff-consolidate.js` funcionando, a proliferação de handoff YAMLs em pipelines longos é gerida automaticamente — sem intervenção manual.

## Risks

- **R1 (MEDIUM):** Hook pode ser invocado em contextos onde git push não é um evento AIOX formal (ex: push de @devops para repo externo). Mitigação: adicionar guard para verificar se o push é para o repo KAIROS_CEREBRO.
- **R2 (LOW):** `settings.json` já pode ter uma entrada para este hook que falhou silenciosamente. Mitigação: verificar entry existente antes de criar.
