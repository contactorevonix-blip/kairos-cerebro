## Summary

Implement validators for Control API workflows. Validate story format (must have acceptance criteria), validate workflow state (must be in correct status before proceeding), prevent invalid agent-story combinations. Validators run before workflow execution.

**Why:** AIOX Constitution (Art. III — Story-Driven) requires strict story validation. Control API must enforce this at boundary.

---

## Acceptance Criteria

- [ ] Create `src/control/validators/StoryValidator.ts`:
  - Checks story exists and is readable
  - Verifies acceptance criteria (not empty, not TBD)
  - Checks story status (must be Draft or Ready for Review before execute)
  - Validates story format (required sections)

- [ ] Create `src/control/validators/WorkflowValidator.ts`:
  - Validates workflow state exists
  - Checks workflow can proceed (no blocking dependencies)
  - Verifies agent assignment matches story requirements
  - Prevents duplicate workflow starts (idempotency)

- [ ] Create `src/control/validators/AgentValidator.ts`:
  - Validates agent exists and is available
  - Checks agent can execute the requested task
  - Prevents agent authority violations (e.g., @dev requesting git push)

- [ ] Error responses:
  ```json
  {
    "error": "ValidationFailed",
    "field": "story_id",
    "reason": "Story must have 3+ acceptance criteria",
    "code": "AC_INSUFFICIENT"
  }
  ```

- [ ] Tests: verify each validator catches violations, happy path passes

---

## Implementation Notes

- **Constitution:** Art. III (Story-Driven) — validators enforce this
- **No invention:** validation rules come from PRD requirements only
- **Fail-fast:** return first validation error, not all
