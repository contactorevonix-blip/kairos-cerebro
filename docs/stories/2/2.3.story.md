# Story 2.3 — Agent Activation Orchestrator

**Epic:** EPIC-2 (Control Core)  
**Status:** Draft  
**Complexity:** 13sp  
**Assignee:** @dev  
**Created:** 2026-06-04

---

## Summary

Build Control API endpoint `POST /workflow/start` that accepts agent + story combo, validates both via Story/Workflow/AgentValidators (Story 2.2), records activation in WORKFLOW-STATE.json, and logs to AuditLog. Enables CLI commands like `kairos-monitor activate @dev story-1.1` to trigger agent workflows programmatically.

**Why:** G4 in PRD requires manual + automated agent orchestration. This is the mechanism that lets CLI + UI activate workflows.

---

## Acceptance Criteria

- [ ] Create `POST /workflow/start` endpoint:
  ```
  POST /workflow/start
  {
    "agent_id": "@dev",
    "story_id": "1.1",
    "auto_start": true  // optional, default false
  }
  ```

- [ ] Response:
  ```json
  {
    "workflow_id": "uuid",
    "agent_id": "@dev",
    "story_id": "1.1",
    "status": "pending",
    "created_at": "ISO8601",
    "message": "Workflow queued. Agent will start when available."
  }
  ```

- [ ] Behavior:
  - Validate agent exists (AgentValidator)
  - Validate story exists (StoryValidator)
  - Check workflow not already running (WorkflowValidator)
  - Create entry in `.aiox/WORKFLOW-STATE.json`
  - Log to AuditLog with `event_type: 'agent_activated'`
  - Return workflow_id immediately (fire-and-forget)

- [ ] Auto-start mechanism:
  - If `auto_start=true`, immediately spawn agent (requires CLI integration, Story 4.1)
  - If `auto_start=false`, queue for manual start

- [ ] Idempotency: same request twice returns same workflow_id
- [ ] Tests: verify validation flow, WORKFLOW-STATE update, audit logging

---

## Implementation Notes

- **Constitution:** Art. II (Agent Authority) — only valid agents can be activated
- **State:** WORKFLOW-STATE.json is source of truth, updated atomically
- **No delay:** response returns immediately, agent starts asynchronously
