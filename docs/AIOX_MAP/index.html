🧬 Core DNA Patterns

1. Agent Specialization

Each agent has exclusive operations. No overlap. Boundaries enforced via authority rules. Examples: @devops only does git push, @pm only creates epics, @sm only drafts stories.

2. Story-Driven Development

No code without story. Stories have acceptance criteria. Progress tracked via checkboxes. File List maintained. Stories are the source of truth for implementation.

3. Quality Gates Non-negotiable

7 mandatory checks: linting, typing, tests, build, CodeRabbit, no secrets, story status. Gates block merge. Self-healing max 2-3 iterations. Escalation forced after max attempts.

4. Authority Hierarchy

Exclusive ops > Delegated ops > Permitted ops. Agents cannot exceed boundary. Violation detection enforced. Escalation to @aiox-master if boundary crossed.

5. Framework Layer Immutability

L1 Core (NEVER modify) — protected by deny rules. L2 Templates (extend-only) — cannot create new templates. L3 Config (conditional mutations) — allow rules permit specific edits. L4 Runtime (always mutable) — full freedom.

6. No Invention Principle

Specs derive only from requirements. Zero invented features. Every statement traces to FR-*/NFR-* or research finding. Constitution Art. IV non-negotiable.

7. IDS Principle (REUSE > ADAPT > CREATE)

Before creating: check existing artefacts. Reuse first. Adapt second. Only create if genuinely new. Prevents duplicate tools, patterns, components.

8. Task-First Execution

Workflows are task chains, not agent chains. Tasks define inputs/outputs/prerequisites. A validated task is law. Tasks are executable, reusable, composable.

🏛️ Framework Boundary (L1-L4)

AIOX uses 4 layers to separate framework artefacts from project runtime.

L1 — Framework Core

Mutability: NEVER modify

Paths: .aiox-core/core/, bin/aiox.js, .aiox-core/constitution.md

Protection: Deny rules (automated enforcement)

Examples: Execution engine, code intelligence, elicitation, events, graph dashboard

L2 — Framework Templates

Mutability: NEVER modify (extend-only)

Paths: .aiox-core/development/tasks/, workflows/, checklists/, .aiox-core/infrastructure/

Protection: Deny rules

Note: Can add NEW tasks but cannot modify existing template structure

L3 — Project Configuration

Mutability: Conditional (allow rules permit)

Paths: .aiox-core/data/, agents/*/MEMORY.md, core-config.yaml

Protection: Allow/deny rules define specific editable sections

Toggle: boundary.frameworkProtection: true/false in config

L4 — Project Runtime

Mutability: ALWAYS modify

Paths: docs/stories/, squads/, tests/, application code

Protection: None — full user freedom

Role: Where actual project work happens
