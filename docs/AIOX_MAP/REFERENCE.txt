                     STORY DEVELOPMENT CYCLE (SDC)
                          Primary Workflow
================================================================================

Phase 1: CREATE (@sm)
  Task: create-next-story
  Input: PRD + epic context
  Output: story.md (Draft status)

Phase 2: VALIDATE (@po)
  Task: validate-next-story
  Input: story.md
  Output: Go/NoGo + 10-point checklist
  Gate: ✅ Must pass validation

Phase 3: IMPLEMENT (@dev)
  Task: dev-develop-story
  Input: validated story + acceptance criteria
  Output: Code + File List
  Branch: feature branch
  Self-critique: Mandatory

Phase 4: QA GATE (@qa)
  Task: qa-gate
  Input: Code
  Output: PASS / CONCERNS / FAIL
  Checks:
    ✓ npm run lint (no errors)
    ✓ npm run typecheck (no errors)
    ✓ npm test (no failures)
    ✓ npm run build (success)
    ✓ CodeRabbit (no CRITICAL issues)
    ✓ No secrets in code
    ✓ Story status is Done

Phase 5: PUSH (@devops)
  Task: git push + gh pr merge
  Authority: @devops EXCLUSIVE
  Process:
    1. Create PR
    2. Merge to main
    3. Tag release (if applicable)

================================================================================
                        DNA MENTAL PATTERNS
================================================================================

1. AGENT SPECIALIZATION
   Each agent has exclusive operations.
   No overlap. Boundaries enforced.
   Examples: @devops only pushes, @pm only creates epics, @sm only drafts

2. STORY-DRIVEN DEVELOPMENT
   No code without story.
   Stories are source of truth.
   Implementation flows from story acceptance criteria.

3. QUALITY GATES NON-NEGOTIABLE
   7 mandatory checks before merge.
   Gates block merge. Self-healing max 2-3 iterations.
   Escalation forced after max attempts.

4. AUTHORITY HIERARCHY
   Exclusive ops > Delegated ops > Permitted ops.
   Agents cannot exceed boundary.
   Violation detection enforced.

5. FRAMEWORK LAYER IMMUTABILITY
   L1-L2 protected by deny rules (cannot modify).
   L3 conditional (allow rules permit specific edits).
   L4 free (always mutable).

6. NO INVENTION PRINCIPLE
   Specs derive only from requirements.
   Zero invented features.
   Every statement traces to requirement or research.

7. IDS PRINCIPLE (REUSE > ADAPT > CREATE)
   Before creating: check existing artefacts.
   Reuse first. Adapt second. Only create if genuinely new.

8. TASK-FIRST EXECUTION
   Workflows are task chains, not agent chains.
   Tasks define inputs/outputs/prerequisites.
   Validated task is law.
