🔄 STORY DEVELOPMENT CYCLE (SDC) — PRIMARY WORKFLOW
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Full 5-phase flow for ALL development work, enforced by gates and tasks.

PHASE 1: CREATE (@sm)
  Task: create-next-story.md
  Trigger: @sm *draft {epic-context}
  Input: PRD sharded, epic structure
  Output: {epicNum}.{storyNum}.story.md (status: Draft)
  ─────────────────────────────────────────────
      ↓

PHASE 2: VALIDATE (@po)
  Task: validate-next-story.md
  Trigger: @po *validate-story-draft {storyId}
  Checklist: 10-point validation
    1. Clear &amp; objective title
    2. Complete description (problem/need)
    3. Testable acceptance criteria (Given/When/Then)
    4. Well-defined scope (IN and OUT)
    5. Dependencies mapped
    6. Complexity estimate (points or T-shirt)
    7. Business value clear
    8. Risks documented
    9. Criteria of Done defined
    10. Alignment with PRD/Epic
  Decision: GO (≥7/10) → Status: Ready | NO-GO → fixes required
  ─────────────────────────────────────────────
      ↓

PHASE 3: IMPLEMENT (@dev)
  Task: dev-develop-story.md
  Trigger: @dev *develop {storyId}
  Execution Modes:
    • YOLO (autonomous, 0-1 prompts)
    • Interactive (5-10 prompts with checkpoints, DEFAULT)
    • Pre-Flight (plan-first, 10-15 prompts then zero-ambiguity)
  Status Transitions:
    Ready → InProgress (when @dev starts)
    InProgress → InReview (when @dev completes)
  CodeRabbit Self-Healing:
    iteration = 0
    while CRITICAL issues found AND iteration < 2:
      auto-fix CRITICAL/HIGH
      iteration++
    if CRITICAL persist after 2 iterations → HALT, manual intervention
  ─────────────────────────────────────────────
      ↓

PHASE 4: QA GATE (@qa)
  Task: qa-gate.md
  Trigger: @qa *qa-gate {storyId}
  7 Quality Checks:
    1. Code review (patterns, readability, maintainability)
    2. Unit tests (coverage, all passing)
    3. Acceptance criteria (all met per story AC)
    4. No regressions (existing functionality preserved)
    5. Performance (within acceptable limits)
    6. Security (OWASP basics verified)
    7. Documentation (updated if necessary)
  Verdicts:
    • PASS → Story status: Done | Proceed to @devops
    • CONCERNS → Story status: Done | Proceed to @devops (with notes)
    • FAIL → Story status: InProgress | Return to @dev with feedback
    • WAIVED → Story status: Done | Approve with waiver documented (rare)
  ─────────────────────────────────────────────
      ↓

PHASE 5: PUSH (@devops)
  Task: pre-push-quality-gate.md
  Trigger: @devops *push {storyId} | *pre-push-quality-gate
  Authority: ONLY after QA PASS/CONCERNS/WAIVED
  Actions:
    ├─ git push (remote)
    ├─ gh pr create (GitHub PR)
    └─ Release management (if applicable)
  ─────────────────────────────────────────────
      ✓ DONE


⚙️  SECONDARY WORKFLOWS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

QA LOOP (Iterative Review-Fix)
  Flow: @qa review → verdict → @dev fixes → re-review (max 5 iterations)
  Commands:
    • *qa-loop {storyId} — Start full loop
    • *qa-loop-review — Resume from review
    • *qa-loop-fix — Resume from fix
    • *escalate-qa-loop — Force manual escalation
  Escalation triggers:
    • max_iterations_reached (5)
    • verdict_blocked
    • fix_failure (after retries)
    • manual_escalate (user command)
  Status tracking: qa/loop-status.json

SPEC PIPELINE (Pre-Implementation Requirements)
  Phases: 3-6 depending on complexity class
  Complexity scoring (1-5 scale each):
    • Scope: files affected
    • Integration: external APIs
    • Infrastructure: changes needed
    • Knowledge: team familiarity
    • Risk: criticality level
  Classes:
    • SIMPLE (score ≤8): 3 phases (gather → spec → critique)
    • STANDARD (score 9-15): 6 phases (full pipeline)
    • COMPLEX (score ≥16): 6 phases + revision cycle
  Constitutional gate (Article IV): Every spec statement MUST trace to
    requirement (FR-*, NFR-*, CON-*) or research finding. NO invention.

BROWNFIELD DISCOVERY (Legacy Assessment)
  10 phases: data collection (1-3) → draft (4-7) → finalization (8-10)
  Outputs: technical-debt-assessment.md + epic for prioritized stories

⚖️  CONSTITUTION — INEGOTIABLE PRINCIPLES
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Article I: CLI FIRST
  Severity: NON-NEGOTIABLE | Gate: Automatic
  All workflows originate in CLI. UI is secondary.
  Impact: Every agent command starts with * prefix

Article II: AGENT AUTHORITY
  Severity: NON-NEGOTIABLE | Gate: Automatic
  Specialized agents own exclusive operations. No cross-boundary violations.
  Violators: System blocks with "agent-authority violation" error

Article III: STORY-DRIVEN DEVELOPMENT
  Severity: MUST | Gate: Enforced at commit time
  Every development task begins with story in docs/stories/
  Story ID must appear in git commit messages and task headers

Article IV: NO INVENTION
  Severity: MUST | Gate: G5 (QA), G6 (CI/CD)
  Spec statements and code MUST trace to requirements.
  Spec pipeline enforces: FR-*, NFR-*, CON-* references mandatory

Article V: QUALITY FIRST
  Severity: MUST | Gate: Automatic at @qa phase
  Quality gates CANNOT be bypassed. CodeRabbit self-healing on CRITICAL/HIGH.
  Waived gates require explicit reason + documentation

Article VI: ABSOLUTE IMPORTS
  Severity: SHOULD | Gate: Informational
  Agents load MEMORY.md from shared knowledge repository.
  Missing MEMORY = graceful fallback to defaults

🎯 PLANNING TRACKS — Scale-Domain-Adaptive

Choose process based on task size and complexity:

QUICK FLOW (< 5 stories, < 2 hours)
  Process: @dev only (no Spec Pipeline)
  When: Bug fixes, config changes, small features
  Workflow: Minimal SDC (skip @pm, @po if trivial)

STANDARD (5-15 stories, 2-5 days)
  Process: Full SDC (@sm → @po → @dev → @qa → @devops)
  When: Feature development, improvements
  Workflow: Complete story lifecycle with gates

ENTERPRISE (> 15 stories, complex/strategic)
  Process: SDC + Spec Pipeline + PRD mandatory
  When: Product launches, epics, transformations
  Workflow: Requirements → Spec → Story → Dev → QA → Push

📋 KEY RULES SUMMARY
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

SMART ROUTING (Before every task)
  1. Diagnose request type
  2. Show routing recommendation
  3. Ask if user has questions
  4. Execute with full context

STORY FILE UPDATES (Permission matrix)
  Who Can Edit:
  • Title, Description, AC, Scope → @po ONLY
  • File List, checkboxes, Dev Notes → @dev
  • QA Results → @qa ONLY
  • Change Log → Any agent (append-only)

CONFIDENCE SCORING (Before execution)
  ≥90%: Execute directly
  70-89%: Present alternatives first
  <70%: Ask clarification questions

IDS DECISION HIERARCHY
  1. REUSE (≥90% relevance) — use directly
  2. ADAPT (60-89% relevance) — modify if adaptability ≥0.6
  3. CREATE (no match) — with full justification &amp; registry entry

TOKEN BUDGET (per task type)
  Simple (typo, config) → 200 tokens
  Medium (bug fix, small feature) → 1,000 tokens
  Complex (new feature, refactor) → 2,500 tokens
